## Architecture

Multi-tenant Flask app. Each league is fully isolated by `league_id`. No data crosses league boundaries.

```
app/
  app.py              # Flask app factory, blueprint registration, context processors
  config.py           # DATABASE_URL, SECRET_KEY env var loading
  database.py         # get_db() ‚Äî branches SQLite vs Postgres; _PgWrapper + _PgCursorWrapper
  init_db.py          # DB init ‚Äî Postgres: runs schema_postgres.sql; SQLite: run_migrations()
  schema_postgres.sql # Full 51-table Postgres schema (source of truth for Postgres)
  routes/             # One blueprint per feature: admin, auth, players, scores, standings, etc.
  templates/          # Jinja2 templates mirroring routes structure
  static/css/main.css
  static/js/main.js
```

Legacy SQLite migration scripts (`migrate_*.py`) live in project root ‚Äî not used under Postgres.

## Conventions

- All queries use `%s` placeholders (psycopg2), never `?` (sqlite3)
- Row access: `row['column_name']` via RealDictCursor ‚Äî never positional index
- INSERTs that need the new PK use `RETURNING <pk>` ‚Äî never `cursor.lastrowid` (returns None in psycopg2)
- Dialect branching where needed: `database.is_postgres()` guards SQLite-only code paths
- `enumerate` registered as both Jinja2 global and filter in app.py
- `login_required` / `admin_required` decorators in auth.py ‚Äî all protected routes use these

## UX Design Patterns

### Score Entry ‚Äî Input Advance Philosophy

Score inputs follow **row-major (player-centric) advance order**: all holes for player 1, then all holes for player 2, etc. This matches the natural DOM tab order (inputs are laid out as player rows √ó hole columns in the HTML).

**Rules:**
- Single digit 1‚Äì9 ‚Üí auto-advance to the next input (same player, next hole). Uses the `input` event, not `keydown` ‚Äî mobile numeric keyboards (iOS/Android) do not reliably fire `keydown` with `e.key` values.
- `Enter` key ‚Üí same advance.
- Typing `0` or any non-single-digit value (e.g. "10") ‚Üí stays in cell. Admin enables "Allow double-digit" for scores ‚â• 10.
- Advance order is built in JS as a flat array (`_scoreAdvanceOrder`) from `players ‚Üí holes` loop nesting (not `holes ‚Üí players`).

**Mobile layout:** Single scrollable table on all screen sizes. The old hole-by-hole card view (with Next/Prev buttons) is retired. Table scrolls horizontally; inputs are 36px wide with `font-size: 16px` to prevent iOS auto-zoom.

**Input type:** Score inputs use `type="text" inputmode="numeric" pattern="[0-9]*"` ‚Äî NOT `type="number"`. On iOS, `type="number"` can return `this.value = ""` during the `input` event before the browser commits the value, which breaks the auto-advance regex check. `type="text"` with `inputmode="numeric"` shows the numeric keyboard on iOS/Android and gives reliable `this.value` on every keystroke. Range validation (1‚Äì20) is enforced server-side.

**iOS focus ‚Äî confirmed working approach:** `input` event + `requestAnimationFrame(() => next.focus())`. Both sync and rAF calls are made (sync for desktop/Android speed; rAF for iOS). Confirmed working on iOS Safari and Chrome iOS as of 2026-06-13.

**What was tried (in order):**
1. `keydown` + `e.key` ‚Äî iOS virtual keyboards don't fire reliable `e.key` values
2. `input` + `type="number"` ‚Äî `this.value` can be `""` during the event on iOS before browser commits
3. `input` + `type="text" inputmode="numeric"` ‚Äî appeared broken; root cause was actually a JS syntax error (see below)
4. `input` + `type="text"` + `requestAnimationFrame(focus)` ‚Äî **confirmed working** ‚úì

**Root cause of all iOS failures:** A missing `</script>` tag before `{% endif %}` in `enter.html` caused a JS syntax error in the holes script block. All holes JS (live calc, stroke dots, auto-advance) was silently dead. The `<script>` for `toggleAbsenceRow` appeared literally inside the unclosed holes `<script>`, and `<script>` is not valid JS syntax. Added `</script>` before `{% endif %}` in commit `37c8fb5`. The iOS focus restriction research was a red herring ‚Äî `input` + rAF focus works fine once the script executes.

**Why player-centric, not hole-centric:**
Walking a golf hole, each player writes down their own score before moving to the next hole. A player-centric tab order (P1: H1‚ÜíH2‚ÜíH3‚Ä¶H9, then P2: H1‚ÜíH2‚Ä¶) matches how a scorer mentally fills the card ‚Äî one player at a time. Hole-centric order (H1: P1‚ÜíP2‚ÜíP3‚ÜíP4, then H2‚Ä¶) adds 17+ unnecessary focus jumps across rows, which feels wrong even on desktop and becomes expensive on mobile.

**Related audit item:** P0-5 in `Audits/2026-06-12-score-entry-audit.md`

**Debug overlay recipe** (add temporarily when JS isn't running, remove after):

## Tee Deduplication

The `tees` table stores one row per (color √ó nine √ó gender). A course with Gold tees can have up to 6 rows: Gold/Front/M, Gold/Back/M, Gold/Full/M, Gold/Front/W, etc. Without dedup, all pickers show 6 separate entries.

**Key columns:** `tee_color` (bare color e.g. "Gold"; may be NULL ‚Äî fall back to `tee_name`), `nine` ("front"/"back"/"full"), `gender` ("M"/"W", default "M").

**Web canonical logic** (`scores.py` ~line 265, used in score entry per-player tee dropdown):
1. Fetch all tees for the course ordered by `gender, tee_name, nine`
2. Determine `selected_nine` from the matchup's assigned `tee_id`
3. Filter to tees where `nine == selected_nine`
4. Group by color (`tee_color` or `tee_name` fallback); prefer M representative over W per color
5. Result: one entry per unique color, tee_id points to M rep for that nine

**API shape for iOS** (`GET /api/v1/courses`, implemented 2026-06-20):
```json
{
  "course_id": 1, "course_name": "Pebble Beach",
  "tee_colors": [
    {
      "color": "Gold",
      "tees": {
        "front": {"tee_id": 10, "tee_name": "Gold (Front)", "holes": [...]},
        "back":  {"tee_id": 11, "tee_name": "Gold (Back)",  "holes": [...]},
        "full":  {"tee_id": 12, "tee_name": "Gold (Full 18)", "holes": [...]}
      }
    }
  ]
}
```

**iOS resolution flow:** Schedule returns `tee_nine` per matchup. Tee picker shows color names only. On submit: `tee_colors[selected_color].tees[matchup.tee_nine].tee_id`.

**Same-color, W-only tees** (no M counterpart) get their own color entry ‚Äî not suppressed.

## Handicap System ‚Äî Chronological Rebuild Engine

*(Added 2026-06-30, WP3.8)*

**Problem this replaces:** the original handicap mechanism (`recalc_handicap_for_player`, `recalc_all_for_season`, the old `_recalc_future_rounds` cascade ‚Äî all still present in `routes/handicap.py`) derives a player's handicap from "whatever the most recent N rounds are right now," via a `handicap_history` query for the latest row. This is correct for the common case (recalculating after the newest round), but breaks for two related cases:
1. Editing/resaving a **past** round overwrites that round's own historical handicap with the player's *current* handicap, instead of the handicap that was correct as of that round's date.
2. The cascade-forward mechanism meant to propagate a correction into later rounds instead flattens every future round to a single "latest" value rather than re-deriving a true chronological progression.

**The fix ‚Äî two-phase chronological replay**, in `routes/handicap.py`:

- **Phase 1 ‚Äî `rebuild_player_handicap_timeline(db, player_id, league_id)`.** Per-player. `handicap_index` is computed purely from a player's own gross-score differentials (window/drop/average), independent of other players or playing handicap ‚Äî so this can run standalone. Walks all of a player's completed rounds oldest‚Üínewest. For each round: looks up *that round's own season's* settings (window size, drop counts, %, max, carry-across-seasons ‚Äî these can vary by season); builds the windowed/dropped/averaged differential pool using only rounds up to and including the current one ‚Äî correctly date-bounded **by construction of the walk**, not via a date-range SQL query; respects manual-override anchors (`handicap_history.is_manual_override=1`, keyed by `trigger_round_id`) by substituting the anchor's value as the entering handicap for the next round instead of the computed average; deletes all auto-computed (`is_manual_override=0`) rows for the player and reinserts them with `calculated_date` set to the round's actual date (not "today"). Returns `{round_id: entering_handicap_index}`.

- **Phase 2 ‚Äî `rebuild_league_handicaps_and_scores(db, league_id)`.** League-wide orchestrator. Runs Phase 1 for every player, merges results into `handicap_lookup = {round_id: {player_id: raw_index}}`. Walks every completed, non-bye matchup in the league in true date order (`round_date ASC, round_id ASC`), calling `_recalc_single_round(..., handicap_lookup=handicap_lookup.get(round_id, {}))` for each ‚Äî re-deriving net scores, A/B roles, and `match_results` using each player's correct point-in-time handicap instead of "whatever's latest". Does not commit ‚Äî caller controls the transaction, which is what makes the admin preview-via-rollback pattern (below) safe.

**`_recalc_single_round` (scores.py)** gained a `handicap_lookup` parameter: when given, `current_handicap(pid)` reads from the lookup instead of querying `handicap_history` for "latest right now". Falls back to the old query when `handicap_lookup` is `None` or doesn't have the player, so existing non-rebuild callers (`recalc_points` admin route, etc.) are unaffected.

**Latent bug fixed as a side effect:** `_recalc_single_round`'s playing-handicap resolution previously only respected a scorecard's `hcp_manually_overridden` flag when `use_existing_hcp=True` was explicitly passed. Cascading callers used `use_existing_hcp=False` by default and would silently overwrite a manually-overridden scorecard's playing handicap during an unrelated cascade. Fixed by checking the override flag unconditionally, before the `use_existing_hcp` branch.

**Call sites wired to the new engine** (the 5 that actually exhibit cascade behavior): `_process_scores` (scores.py, after every round save), `matrix_update`, `clear_scorecard_overrides`, `clear_handicap_override` (handicap.py), and the league-settings-save cascade (admin.py ‚Äî also now re-derives the index using just-changed settings, not just re-scoring with stale `handicap_history` values).

**Deliberately left on the legacy mechanism** (single-shot recalc, no cascade, so the bug doesn't apply): `score_import.py`, `self_report.py`, `api.py` (4 call sites), `migration.py`. If any of these later need cascade behavior, migrate them to `rebuild_league_handicaps_and_scores` the same way.

**`recalc_season` (the existing "‚Ü∫ Recalc Handicaps" button)** was deliberately left unchanged ‚Äî narrower, season-scoped, handicap-index-only tool, kept distinct from the new league-wide rebuild.

**Admin UI ‚Äî `GET/POST /handicap/rebuild`:** GET runs `rebuild_league_handicaps_and_scores` against the live connection then calls `db.rollback()` ‚Äî a true preview sharing the exact same code path as the real apply, not a separate simulation. POST runs the same call and commits. Template: `templates/handicap/rebuild_timeline.html`. Entry point: Admin ‚Üí More Tools ‚Üí üõÝ Rebuild Handicap Timeline.

**Two override mechanisms, don't conflate them:**
- `scorecards.hcp_manually_overridden` ‚Äî per-round **playing handicap** override (independent of the index timeline; e.g. matrix edits).
- `handicap_history.is_manual_override` ‚Äî global **index timeline anchor**, tied to `trigger_round_id`.

Since differentials (which drive the index) are computed from gross scores only, per-round playing-handicap overrides never affect the index timeline ‚Äî Phase 1 and Phase 2 are cleanly decoupled.

**Known limitation:** this code has not been run against live data (no DB credentials available in the build environment) ‚Äî validated by `py_compile` and code review only. @user must run the preview and apply via the admin route, and spot-check at least the originally-reported matchup and a multi-season/override player.

## Absence / Ghost-Score Handling

*(Added 2026-06-30, WP3.9)*

A `scorecards.is_absent = 1` row marks an absent (no-sub) player whose hole scores are server-synthesized "ghost scores" (par + handicap strokes), not actually played. Two independent concerns follow from this, governed differently ‚Äî don't conflate them:

- **Match points** (`match_results.total_points` / `overall_point_won`) ‚Äî whether a ghost score counts as a win for overall points is an admin policy choice, not a stats-display choice. Controlled by the `absence_overall_point_policy` setting (`overall_wins` / `never_wins` / `excused_only`, default `excused_only` ‚Äî matches pre-existing behavior). Read via `_settings_absence_policy()` (scores.py), applied via the shared `_apply_absence_overall_policy()` helper. Never filter `is_absent` out of a query purely to "fix" points ‚Äî that's the policy setting's job.
- **Gross/hole-level stats** (avg gross, best gross, eagle/birdie counts, handicap differentials, rounds-played-for-gross-purposes) ‚Äî ghost scores are not real performance and should always be excluded, unconditionally, regardless of the points policy. Visually reinforced with the `sc-player-row--ghost` CSS class (italicized hole scores) across scorecard/entry templates.

**The gotcha:** several queries compute both kinds of stat from the *same row set* in one SQL statement (e.g. a player's round history joining `scorecards` for gross totals while also summing `mr.total_points`). Adding a blanket `AND sc.is_absent = 0` to the `WHERE` clause is only safe when the query is gross-only. If a points field is also being aggregated, a blanket filter silently zeroes out that player's correctly-policy-governed points for absence weeks ‚Äî a real regression, not a stats fix. Fix pattern: keep the `WHERE` clause unfiltered and use conditional aggregation instead, e.g.:
```sql
COUNT(DISTINCT CASE WHEN sc.is_absent = 0 THEN sc.scorecard_id END) AS rounds_played,
COALESCE(SUM(CASE WHEN sc.is_absent = 0 THEN gross_totals.total_gross ELSE 0 END), 0) AS total_gross,
COALESCE(SUM(mr.total_points), 0) AS total_pts  -- left unfiltered
```
or, in Python, carry `sc.is_absent` through the SELECT and filter only the gross-derived lists after the fact (e.g. `players.py`'s `round_rows`/`_season_stats`). Applied this way in `handicap.py`, `players.py`, `stats.py`, `my_stats.py`, `records.py`. Deliberately **not** applied to `skins.py` ‚Äî skins participation is a manual per-round opt-in checkbox, not an automated stats aggregate, so an admin already controls absent-player eligibility there.

## Known Constraints / Gotchas

- **Supabase port:** Use connection pooler port `6543`, not direct `5432` ‚Äî direct host is IPv6-only and fails with "Network is unreachable" on Render
- **Postgres GROUP BY strictness:** SQLite allows selecting non-aggregate columns not in GROUP BY; Postgres does not. Every non-aggregate SELECT column must appear in GROUP BY. Large source of migration bugs ‚Äî see `migration.md` Phase 5.3 for the full list of files fixed.
- **HAVING aliases:** Postgres `HAVING` cannot reference SELECT-list aliases. Replace `HAVING alias >= N` with the actual aggregate expression.
- **Dynamic `?` placeholders:** The tokenize-based `?`‚Üí`%s` script missed placeholders built via string concatenation (`query += " AND x = ?"`). Three files had these: `handicap.py`, `players.py`, `stats.py` ‚Äî all fixed.
- **`ensure_tables()` in admin.py:** SQLite-only DDL; guarded with `DATABASE_URL` check ‚Äî no-ops under Postgres. Tables come from `schema_postgres.sql`.
- **Render free tier spins down** after inactivity ‚Äî upgrade to $7/mo paid before sharing demo URL publicly.
- **Mixed gross-stats/points queries and `is_absent`:** see "Absence / Ghost-Score Handling" above ‚Äî never blanket-filter `is_absent` out of a query that also aggregates `match_results.total_points`.
- **`division` vs `division_name` column:** `schema_postgres.sql` uses `division_name`; old queries selected `division` ‚Üí `UndefinedColumn`. Fixed in `league_info.py`.
