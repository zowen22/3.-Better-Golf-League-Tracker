## Architecture

Multi-tenant Flask app. Each league is fully isolated by `league_id`. No data crosses league boundaries.

```
app/
  app.py              # Flask app factory, blueprint registration, context processors
  config.py           # DATABASE_URL, SECRET_KEY env var loading
  database.py         # get_db() — branches SQLite vs Postgres; _PgWrapper + _PgCursorWrapper
  init_db.py          # DB init — Postgres: runs schema_postgres.sql; SQLite: run_migrations()
  schema_postgres.sql # Full 51-table Postgres schema (source of truth for Postgres)
  routes/             # One blueprint per feature: admin, auth, players, scores, standings, etc.
  templates/          # Jinja2 templates mirroring routes structure
  static/css/main.css
  static/js/main.js
```

Legacy SQLite migration scripts (`migrate_*.py`) live in project root — not used under Postgres.

## Conventions

- All queries use `%s` placeholders (psycopg2), never `?` (sqlite3)
- Row access: `row['column_name']` via RealDictCursor — never positional index
- INSERTs that need the new PK use `RETURNING <pk>` — never `cursor.lastrowid` (returns None in psycopg2)
- Dialect branching where needed: `database.is_postgres()` guards SQLite-only code paths
- `enumerate` registered as both Jinja2 global and filter in app.py
- **Admin League Settings numbering** (`admin/settings.html`, added 2026-07-02): every section header and field label is prefixed with an `X.XX` reference number via `<span class="setting-num">`, e.g. `2.05` = Handicap Percent. First digit = section (1 Scoring, 2 Handicap, 3 Max Score Per Hole, 4 Playoffs, 5 Skins Defaults, 6 Self-Reporting, 7 Season Segments, 8 Tiebreakers), two-digit suffix = field order within that section. When adding a new setting, number it as the next `X.XX` in its section (or start a new section number at the end if it doesn't fit an existing category) — don't renumber existing fields, since these numbers are meant to be stable references (e.g. in support conversations, docs).
- `login_required` / `admin_required` decorators in auth.py — all protected routes use these

Idempotent — safe to re-run.

**Bugs found and fixed getting local dev working for the first time (2026-07-03)** — all four were latent, never caught because the app had apparently never been run against a genuinely fresh Postgres DB before (only ever exercised against production Supabase, which already had years of accumulated manual/ad-hoc state):
1. `init_db.py _seed_if_empty()`: used the raw SQLite connection object instead of a cursor (`cur.fetchone()` on a `Connection` has no such method) — crashed on the very first empty-DB check.
2. `schema_postgres.sql`: `courses.default_tee_id` had an inline `FOREIGN KEY ... REFERENCES tees(tee_id)`, but `tees` isn't defined until after `courses` in the file — a genuine circular dependency (`tees.course_id` also references `courses`) that only breaks on a from-scratch DB. Fixed by moving that one FK to a post-creation `ALTER TABLE ... ADD CONSTRAINT` (idempotency-guarded via `pg_constraint` check, since this file re-runs on every app startup).
3. Two `DATE_TRUNC('month', requested_at)` indexes (one in `schema_postgres.sql`, one in the now-redundant `migrations/add_course_api_cache.sql`) failed with "functions in index expression must be marked IMMUTABLE" — `DATE_TRUNC` on a `TIMESTAMPTZ` column is STABLE, not IMMUTABLE, since it depends on the session timezone. Removed both (the underlying rate-limit query still works without them, just without that specific optimization on a low-volume internal table).
4. `_apply_additive_migrations_postgres()`'s migration loop ran every file against the same cursor/transaction with no per-file isolation — one failing migration (the DATE_TRUNC one above) poisoned every migration after it in the list ("current transaction is aborted, commands ignored until end of transaction block"), silently skipping all of them. Fixed with a `SAVEPOINT`/`ROLLBACK TO SAVEPOINT` around each migration so a single failure can't cascade.
5. **Registered 5 migrations that existed as files but were never in the additive list** — the exact "written but never registered" failure class already documented below, just newly discovered instances of it: `add_is_sub.sql` (`players.is_sub` — missing entirely broke score entry with `UndefinedColumn`), `add_absence_sub_name.sql`, `add_round_reflections.sql`, `add_scorecard_hcp_override.sql` (**`scorecards.hcp_manually_overridden`** — the column this entire session's absence/override work depends on, missing from a fresh DB), `nullable_absence_round_id.sql`. The other 9 unregistered migration files were checked and found genuinely redundant (their column/table already exists in `schema_postgres.sql` from being folded in later) — left unregistered with an explanatory comment rather than blindly re-added.
6. **`rebuild_league_handicaps_and_scores()`'s Phase 2 silently skipped every round in any season with no `league_settings` row** (`get_league_settings()` returns raw `None`, and the Phase 2 loop treated that as `continue` instead of falling back to the same defaults every other call site already uses). Real production risk, not just a seed-script artifact: `seasons.create()` never inserts a `league_settings` row — only the first Admin Settings save does — so any brand-new season nobody has visited Settings for yet would have this exact "Rebuild Handicap Timeline silently does nothing" gap. Fixed by falling back to the standard defaults (90.0/18.0/'match_play'/'excused_only') instead of skipping. Caught by seeding a fresh season and finding a pre-eligibility round's playing handicap stayed at its naive initial-save value (0) instead of correcting to the temp handicap (18) after the rebuild ran — `rounds_changed: 0` even on a from-scratch rebuild was the tell.

**Two more downstream instances of the same `seasons.create()`/`league_settings` root gap found and fixed 2026-07-03** (see `Audits/2026-07-03-league-creation-new-season-audit.md`): (a) `get_league_settings(db, season_id, league_id)` (`scores.py:146`) was called with the two ID args **swapped** at `admin.py:163` (`panel()`) and `schedule.py:2022` (`week_scorecards()`) — only "worked" by coincidence for a league's first season (`league_id == season_id`), silently returned `None`/hardcoded defaults for any season after that, with no other call site affected (all ~15 others already correctly ordered). (b) `schedule.py:_get_single_course()` treated a missing `league_settings` row the same as `multi_course=1`, so generating a schedule before the admin's first Settings save gave every matchup `NULL` course_id/tee_id even for a genuinely single-course league — fixed to treat a missing row as `multi_course=0` (its actual schema default), matching the Phase-2 fallback precedent above. The root gap itself (`seasons.create()` never writing a `league_settings` row) was deliberately left unfixed — whether to auto-write defaults there, or ship a "copy previous season's settings/teams" feature instead, is a real product decision, not a bug; see the audit doc's Recommendations.

## Billing (Stripe) — Recurring Annual Subscription, Web-Only

**Built 2026-07-15.** One Stripe subscription per league (`subscriptions` table, `league_id` UNIQUE), $99/year via Stripe Checkout, 14-day trial (`STRIPE_TRIAL_DAYS`). `subscriptions` mirrors Stripe's own state via webhooks — it doesn't compute status independently, so it can never drift into a state Stripe itself doesn't agree with.

**Why no StoreKit / Apple In-App Purchase**: verified directly against Apple's current App Store Review Guidelines (3.1.3(b), fetched live 2026-07-15, not from training-data memory) — an app may let users access a subscription purchased on the web, as long as the app itself has no purchase button or external checkout link. BGLT's iOS app never sells anything (league creation/billing is web-only), so this applies cleanly — it's the same pattern Netflix/Kindle use, not a guideline workaround. This resolves the "cross-platform entitlement" open question from 2026-07-03: there's only ever one entitlement source (Stripe), so nothing to reconcile across platforms.

**Files**: `routes/billing.py` (blueprint, `/billing` prefix), `templates/billing/`, `migrations/add_subscriptions.sql` (registered in `init_db.py`'s additive list + folded into `schema_postgres.sql` directly). Config in `config.py`: `STRIPE_SECRET_KEY`, `STRIPE_PUBLISHABLE_KEY`, `STRIPE_WEBHOOK_SECRET`, `STRIPE_PRICE_ID_ANNUAL`, `STRIPE_TRIAL_DAYS` — all unset in dev by design, `billing.py` degrades to a "not configured" message rather than erroring.

**Routes**: `/billing` (status + Subscribe/Manage buttons, `admin_required`), `/billing/checkout` (creates a Stripe Checkout Session, redirects), `/billing/portal` (Stripe Customer Portal for self-service cancel/manage), `/billing/webhook` (signature-verified via `STRIPE_WEBHOOK_SECRET`, CSRF-exempted like `api_bp` — handles `checkout.session.completed`, `customer.subscription.updated/created/deleted`, `invoice.payment_failed`).

**Deliberately NOT built this pass — infrastructure only, no enforcement**: no route anywhere checks `has_active_subscription()` (provided in `billing.py` for whichever future pass wires it in). Turning on paywall enforcement is a separate decision that needs explicit sign-off — the real production Root Beer League (and any other league already live) has no subscription today and would be locked out the instant enforcement went live. Site Admin Dashboard got a new "Billing" widget (trial/active/past_due/canceled/no-subscription counts across all leagues) — read-only, same pattern as its existing API-usage widgets, wrapped in `try/except` since `subscriptions` won't exist on production until that migration is actually run there.

**What's still needed before this can go live**: a real Stripe account (@user's to create — requires business/bank details Claude can't provide), the env vars above set on Render, and the migration run against production Supabase. Volume pricing ($79/league at 10+ leagues) has no attachment mechanism yet — it requires a customer/organization spanning multiple leagues, and multi-league user accounts are still explicitly deferred (Project Overview's Scope: Out) — single-league billing at $99/year is what's actually implemented.

## Season Context & Rollover

*(Added 2026-07-03, WP3.20)*

**Current-season model:** the app "lives" inside one season at a time. `get_current_season_id(db, league_id)` (`database.py`) is the single source of truth — it returns `session['current_season_id']` after validating that season belongs to the current league (guards stale ids after a league switch), else falls back to newest-by-`season_id` and writes the session key. `inject_nav_context()` (`app.py`) and every "current-season" page route use it; the nav drawer's "Switch Season" select (`/switch-season/<id>`) therefore moves the *whole app* into the chosen season, not just the dropdown. When adding a new route that needs "the current season," call this helper — do **not** write another `ORDER BY season_id DESC LIMIT 1`. Known deliberate exceptions: `admin.overview()` (has its own season-picker UI), `api.py`'s `_current_season()` (mobile JWT, no browser session), `public_view.py` (unauthenticated).

**Year-prefixed headers:** `inject_nav_context()` exposes `nav_season_year` (first 4 chars of the current season's `start_date` if `YYYY…`, else a 19xx/20xx token in `season_name`, else None) — rendered as `2026 · League Name` in the drawer header and dashboard page header. The "Start Another Season" wizard requires `start_date` partly to feed this.

**"Start Another Season" (client-facing verbiage — never "copy previous season"):** wizard at `/seasons/start-next` (`seasons.py`). Single-transaction create with opt-out bring-overs: teams & divisions (straight re-point to new `season_id`, inactive-player teams copied and flagged in the hub, not dropped), league settings (explicit clone column list `_LEAGUE_SETTINGS_CLONE_COLUMNS` — keep in sync when adding `league_settings` columns! — `dues_amount` kept, `dues_due_date` forced NULL), and starting-handicap seeding via the shared `_seed_starting_handicaps(db, league_id, season_id)` (`admin.py`, also used by the standalone seed route; no commit inside, caller owns the transaction). Players/courses/tees are league-scoped and always persist — nothing to copy. On success the session moves into the new season and redirects to the **Season Setup hub** (`/seasons/<id>/setup`): status cards for roster, player activation, settings, schedule, buy-ins (reuses `dues.py`'s paid/eligible derivation), and starting handicaps.

**Season-over signal:** `season_is_over(db, season_id)` (`seasons.py`) — `MAX(matchups.scheduled_date) < today` (ISO-string compare), falling back to `end_date` when no matchups. Drives the dashboard admin CTA card and the admin-panel banner; a "Continue season setup" link appears when a newer season still lacks matchups or settings. Note the import direction: `seasons.py` imports `_seed_starting_handicaps` from `admin.py` at module level, so `admin.py` imports `season_is_over` lazily inside `panel()` to avoid the cycle.

## UX Design Patterns

### Score Entry — Input Advance Philosophy

Score inputs follow **row-major (player-centric) advance order**: all holes for player 1, then all holes for player 2, etc. This matches the natural DOM tab order (inputs are laid out as player rows × hole columns in the HTML).

**Rules:**
- Single digit 1–9 → auto-advance to the next input (same player, next hole). Uses the `input` event, not `keydown` — mobile numeric keyboards (iOS/Android) do not reliably fire `keydown` with `e.key` values.
- `Enter` key → same advance.
- Typing `0` or any non-single-digit value (e.g. "10") → stays in cell. Admin enables "Allow double-digit" for scores ≥ 10.
- Advance order is built in JS as a flat array (`_scoreAdvanceOrder`) from `players → holes` loop nesting (not `holes → players`).

**Mobile layout (unified 2026-07-03):** `scores/enter.html` and `scores/enter_week.html` now share one mobile scorecard treatment, per `Audits/2026-07-03-mobile-entry-unification.md`:
- **Part 1 — dead card view removed.** The old hole-by-hole card view (markup, controller JS — `loadHole()`/`syncAndCalc()`/`window.updateMobileView` — and the `.mobile-sc-section`/`.mob-*` CSS family) was fully deleted from `enter.html`, along with the `.mob-score-input` special case in `base.html`'s `spApplyAbsenceToScores`. It had been visually retired (`display: none !important`, no media query ever re-showed it) but was still shipping and executing against hidden elements — pure dead weight, zero user-visible change from removing it.
- **Part 2 — shared compact table.** Both pages now use the same `@media (max-width: 640px)` compact table (`main.css`, in the "COMPACT MOBILE SCORECARD TABLE" section) and the same `fit()` JS (moved from `enter_week.html` into `base.html` so it's shared — page-agnostic, operates on every `.scorecard-table` in the DOM via `document.querySelectorAll`, no-ops harmlessly above 640px or where there's no scorecard). `fit()` computes per-hole column widths via canvas `measureText` once fonts are ready, and on width-changing resizes; names are abbreviated to first-name-only on mobile via `.sc-name-first`/`.sc-name-last` (last name hidden) so `fit()` can measure a shorter string.
- **Completed-view Pts column:** the Tot/Hcp/Net/Pts summary columns are hidden during live entry and revealed via a shared `sc-completed-view` class (renamed from `enter_week.html`'s old `ew-completed-scorecard`) applied to the `.scorecard-wrap` div — `enter_week.html` applies it to its locked/completed matchup cards; `enter.html` applies it via `{% if matchup['status'] == 'completed' %}` since it always renders one live editable form regardless of status (no separate locked view).
- **Shared Hcp-marker overlay CSS** (`.hcp-cell-wrap`/`.hcp-provisional-mark` positioning, both the desktop base rule and the mobile override) was folded into `main.css`, replacing byte-identical copies that had drifted into both templates' local `<style>` blocks — moving only the *mobile override* to main.css while leaving the *base* rule duplicated locally would have silently broken the override (main.css loads before either template's local `<style>`, so an unconditional local rule of equal specificity would win at every width via cascade source order).
- Breakpoint reconciled to 640px as the primary mobile breakpoint (matching `enter_week.html`'s original, better-tested treatment); the remaining 641-768px `main.css` block now only sets `.scorecard-wrap` horizontal-scroll + `font-size: 16px` anti-zoom on `.score-input` (the old fixed-36px-width override was removed as redundant/conflicting with the new fixed-layout table).

**Input type:** Score inputs use `type="text" inputmode="numeric" pattern="[0-9]*"` — NOT `type="number"`. On iOS, `type="number"` can return `this.value = ""` during the `input` event before the browser commits the value, which breaks the auto-advance regex check. `type="text"` with `inputmode="numeric"` shows the numeric keyboard on iOS/Android and gives reliable `this.value` on every keystroke. Range validation (1–20) is enforced server-side.

**iOS focus — confirmed working approach:** `input` event + `requestAnimationFrame(() => next.focus())`. Both sync and rAF calls are made (sync for desktop/Android speed; rAF for iOS). Confirmed working on iOS Safari and Chrome iOS as of 2026-06-13.

**What was tried (in order):**
1. `keydown` + `e.key` — iOS virtual keyboards don't fire reliable `e.key` values
2. `input` + `type="number"` — `this.value` can be `""` during the event on iOS before browser commits
3. `input` + `type="text" inputmode="numeric"` — appeared broken; root cause was actually a JS syntax error (see below)
4. `input` + `type="text"` + `requestAnimationFrame(focus)` — **confirmed working** ✓

**Root cause of all iOS failures:** A missing `</script>` tag before `{% endif %}` in `enter.html` caused a JS syntax error in the holes script block. All holes JS (live calc, stroke dots, auto-advance) was silently dead. The `<script>` for `toggleAbsenceRow` appeared literally inside the unclosed holes `<script>`, and `<script>` is not valid JS syntax. Added `</script>` before `{% endif %}` in commit `37c8fb5`. The iOS focus restriction research was a red herring — `input` + rAF focus works fine once the script executes.

**Why player-centric, not hole-centric:**
Walking a golf hole, each player writes down their own score before moving to the next hole. A player-centric tab order (P1: H1→H2→H3…H9, then P2: H1→H2…) matches how a scorer mentally fills the card — one player at a time. Hole-centric order (H1: P1→P2→P3→P4, then H2…) adds 17+ unnecessary focus jumps across rows, which feels wrong even on desktop and becomes expensive on mobile.

**Related audit item:** P0-5 in `Audits/2026-06-12-score-entry-audit.md`

## Tee Deduplication

The `tees` table stores one row per (color × nine × gender). A course with Gold tees can have up to 6 rows: Gold/Front/M, Gold/Back/M, Gold/Full/M, Gold/Front/W, etc. Without dedup, all pickers show 6 separate entries.

**Key columns:** `tee_color` (bare color e.g. "Gold"; may be NULL — fall back to `tee_name`), `nine` ("front"/"back"/"full"), `gender` ("M"/"W", default "M").

**Web canonical logic** (`scores.py` ~line 265, used in score entry per-player tee dropdown):
1. Fetch all tees for the course ordered by `gender, tee_name, nine`
2. Determine `selected_nine` from the matchup's assigned `tee_id`
3. Filter to tees where `nine == selected_nine`
4. Group by color (`tee_color` or `tee_name` fallback); prefer M representative over W per color
5. Result: one entry per unique color, tee_id points to M rep for that nine

**API shape for iOS** (`GET /api/v1/courses`, implemented 2026-06-20):
```json
{
  "course_id": 1, "course_name": "Pebble Beach",
  "tee_colors": [
    {
      "color": "Gold",
      "tees": {
        "front": {"tee_id": 10, "tee_name": "Gold (Front)", "holes": [...]},
        "back":  {"tee_id": 11, "tee_name": "Gold (Back)",  "holes": [...]},
        "full":  {"tee_id": 12, "tee_name": "Gold (Full 18)", "holes": [...]}
      }
    }
  ]
}
```

**iOS resolution flow:** Schedule returns `tee_nine` per matchup. Tee picker shows color names only. On submit: `tee_colors[selected_color].tees[matchup.tee_nine].tee_id`.

**Same-color, W-only tees** (no M counterpart) get their own color entry — not suppressed.

## Handicap System — Chronological Rebuild Engine

*(Added 2026-06-30, WP3.8)*

**Problem this replaces:** the original handicap mechanism (`recalc_handicap_for_player`, `recalc_all_for_season`, the old `_recalc_future_rounds` cascade — all still present in `routes/handicap.py`) derives a player's handicap from "whatever the most recent N rounds are right now," via a `handicap_history` query for the latest row. This is correct for the common case (recalculating after the newest round), but breaks for two related cases:
1. Editing/resaving a **past** round overwrites that round's own historical handicap with the player's *current* handicap, instead of the handicap that was correct as of that round's date.
2. The cascade-forward mechanism meant to propagate a correction into later rounds instead flattens every future round to a single "latest" value rather than re-deriving a true chronological progression.

**The fix — two-phase chronological replay**, in `routes/handicap.py`:

- **Phase 1 — `rebuild_player_handicap_timeline(db, player_id, league_id)`.** Per-player. `handicap_index` is computed purely from a player's own gross-score differentials (window/drop/average), independent of other players or playing handicap — so this can run standalone. Walks all of a player's completed rounds oldest→newest. For each round: looks up *that round's own season's* settings (window size, drop counts, %, max, carry-across-seasons — these can vary by season); builds the windowed/dropped/averaged differential pool using only rounds up to and including the current one — correctly date-bounded **by construction of the walk**, not via a date-range SQL query; respects manual-override anchors (`handicap_history.is_manual_override=1`, keyed by `trigger_round_id`) by substituting the anchor's value as the entering handicap for the next round instead of the computed average; deletes all auto-computed (`is_manual_override=0`) rows for the player and reinserts them with `calculated_date` set to the round's actual date (not "today"). Returns `{round_id: entering_handicap_index}`.

- **Phase 2 — `rebuild_league_handicaps_and_scores(db, league_id)`.** League-wide orchestrator. Runs Phase 1 for every player, merges results into `handicap_lookup = {round_id: {player_id: raw_index}}`. Walks every completed, non-bye matchup in the league in true date order (`round_date ASC, round_id ASC`), calling `_recalc_single_round(..., handicap_lookup=handicap_lookup.get(round_id, {}))` for each — re-deriving net scores, A/B roles, and `match_results` using each player's correct point-in-time handicap instead of "whatever's latest". Does not commit — caller controls the transaction, which is what makes the admin preview-via-rollback pattern (below) safe.

**`_recalc_single_round` (scores.py)** gained a `handicap_lookup` parameter: when given, `current_handicap(pid)` reads from the lookup instead of querying `handicap_history` for "latest right now". Falls back to the old query when `handicap_lookup` is `None` or doesn't have the player, so existing non-rebuild callers (`recalc_points` admin route, etc.) are unaffected.

**Latent bug fixed as a side effect:** `_recalc_single_round`'s playing-handicap resolution previously only respected a scorecard's `hcp_manually_overridden` flag when `use_existing_hcp=True` was explicitly passed. Cascading callers used `use_existing_hcp=False` by default and would silently overwrite a manually-overridden scorecard's playing handicap during an unrelated cascade. Fixed by checking the override flag unconditionally, before the `use_existing_hcp` branch.

**Call sites wired to the new engine** (the 5 that actually exhibit cascade behavior): `_process_scores` (scores.py, after every round save), `matrix_update`, `clear_scorecard_overrides`, `clear_handicap_override` (handicap.py), and the league-settings-save cascade (admin.py — also now re-derives the index using just-changed settings, not just re-scoring with stale `handicap_history` values).

**Deliberately left on the legacy mechanism** (single-shot recalc, no cascade, so the bug doesn't apply): `score_import.py`, `self_report.py`, `api.py` (4 call sites), `migration.py`. If any of these later need cascade behavior, migrate them to `rebuild_league_handicaps_and_scores` the same way.

## Absence / Ghost-Score Handling

*(Added 2026-06-30, WP3.9)*

A `scorecards.is_absent = 1` row marks an absent (no-sub) player whose hole scores are server-synthesized "ghost scores" (par + handicap strokes), not actually played. Two independent concerns follow from this, governed differently — don't conflate them:

- **Match points** (`match_results.total_points` / `overall_point_won`) — whether a ghost score counts as a win for overall points is an admin policy choice, not a stats-display choice. Controlled by the `absence_overall_point_policy` setting (`overall_wins` / `never_wins` / `excused_only`, default `excused_only` — matches pre-existing behavior). Read via `_settings_absence_policy()` (scores.py), applied via the shared `_apply_absence_overall_policy()` helper. Never filter `is_absent` out of a query purely to "fix" points — that's the policy setting's job.
- **Gross/hole-level stats** (avg gross, best gross, eagle/birdie counts, handicap differentials, rounds-played-for-gross-purposes) — ghost scores are not real performance and should always be excluded, unconditionally, regardless of the points policy. Visually reinforced with the `sc-player-row--ghost` CSS class (italicized hole scores) across scorecard/entry templates.

**The gotcha:** several queries compute both kinds of stat from the *same row set* in one SQL statement (e.g. a player's round history joining `scorecards` for gross totals while also summing `mr.total_points`). Adding a blanket `AND sc.is_absent = 0` to the `WHERE` clause is only safe when the query is gross-only. If a points field is also being aggregated, a blanket filter silently zeroes out that player's correctly-policy-governed points for absence weeks — a real regression, not a stats fix. Fix pattern: keep the `WHERE` clause unfiltered and use conditional aggregation instead, e.g.:
```sql
COUNT(DISTINCT CASE WHEN sc.is_absent = 0 THEN sc.scorecard_id END) AS rounds_played,
COALESCE(SUM(CASE WHEN sc.is_absent = 0 THEN gross_totals.total_gross ELSE 0 END), 0) AS total_gross,
COALESCE(SUM(mr.total_points), 0) AS total_pts  -- left unfiltered
```
or, in Python, carry `sc.is_absent` through the SELECT and filter only the gross-derived lists after the fact (e.g. `players.py`'s `round_rows`/`_season_stats`). Applied this way in `handicap.py`, `players.py`, `stats.py`, `my_stats.py`, `records.py`. Deliberately **not** applied to `skins.py` — skins participation is a manual per-round opt-in checkbox, not an automated stats aggregate, so an admin already controls absent-player eligibility there.

**Second sweep (2026-07-02):** a ghost/handicap-override interaction corrupted a real player's Net (see "Ghost Score + Manual Override Interaction" below), which prompted a full league-wide re-audit via 3 parallel Explore agents. Found 15 more missed call sites — `reports.py`, `my_stats.py`, `players.py` (2 each), `standings.py` (5), `api.py` (3), `schedule.py` (3) — all fixed with the same `AND sc.is_absent = 0` / conditional-aggregation pattern above. The gap existed because the pattern was applied file-by-file as each stats page was touched, with no single audit ever covering every file at once — if a new stats query is added anywhere, it needs this filter explicitly; there's no shared query builder that applies it automatically.

**Ghost Score + Manual Override Interaction (2026-07-02):** a ghost's synthesized gross (`par + strokes_on_hole(ph, ...)`) is only correct for the `ph` (playing handicap) it was generated under. Any code path that re-derives Net from a *stored* ghost gross using a handicap value fetched *separately* — rather than regenerating gross and net together from the same current handicap — will silently corrupt Net the moment the handicap changes after the ghost was written (e.g. via a Handicap Matrix override). This bit `admin.py`'s "Edit Scores" tool: it pre-filled a ghost's stored gross into an editable form, then combined it with whatever handicap is stored *now* to compute Net on save. Fixed by making that tool ghost-aware (regenerate gross + set net = par fresh, ignore any submitted ghost value) rather than patching it as a one-off. **Rule of thumb:** any code that touches a ghost's gross or net must always derive *both* from the *same* handicap value in the *same* operation — never trust one to already be consistent with the other from an earlier write.

## Match Play Scoring — Absolute Net vs. Differential Net

*(Added 2026-07-02, WP3.14)*

Two different "net score" concepts coexist in match-play mode, and they drive **different** parts of the score — this is not a bug, it's the design:

- **Absolute net** (`hole_scores.net_score`, stored in the DB) — each player's own gross score minus strokes from their *own* full playing handicap against par. Personal, opponent-independent. Drives: personal stats, the "Net" column/total on the Scorecards page, Standings, the handicap-history/differential calculation, **and the overall point** (a stroke-play-style "who shot the better net round" comparison — `calc_match_play(sum(net_x), sum(net_y))`). Never changes based on who the opponent is.
- **Differential net** (computed on the fly, never stored) — only the *higher-handicap* player in a pair receives strokes, equal to the handicap gap between the two players, allocated to the hardest holes by `handicap_index` rank. This is what real singles match play uses to decide who wins a hole. Drives: **hole-by-hole win/loss/tie only**. Depends on the opponent — the same round can have a different differential-net result depending on who's paired against whom.

**Why they can disagree:** absolute-net comparison and differential-net comparison are not algebraically equivalent, because `strokes_on_hole()` allocates a 2nd stroke once playing handicap exceeds `total_holes + rank`. Concrete example: player A (hcp 20) vs player B (hcp 8), hole rank 1, 9 holes — absolute method gives A 2 strokes and B 1 stroke (net swing of 1); differential method gives A the full 12-stroke gap allocated by rank, which nets to a 2-stroke swing on that hole. They only coincide when neither player crosses the "2nd stroke" threshold.

**The bug (fixed 2026-07-02) and its correction (same day):** every match-play hole/overall-point calculation across the codebase was using absolute-net comparison for everything. First pass switched both hole points and the overall point to differential — but the overall point is meant to stay absolute (stroke-play-style, not match-play-style); only hole-by-hole win/loss should use differential. `diff_match_hole_points(gross_x, gross_y, holes_x, ph_x, ph_y, net_x, net_y)` in `scores.py` implements the corrected split: the per-hole loop uses `ph_x`/`ph_y` (differential), the return's overall figure sums `net_x`/`net_y` (absolute, the same lists already used to write `hole_scores.net_score`). Applied to all 7 write paths (`_process_scores`, `_recalc_single_round`, `admin.py _save_edited_scores`, `api.py` x2, `self_report.py`, `score_import.py`) plus the display-only duplicate that computes its own overall figure (`debug_scores.py build_pair`) and the client-side JS live preview (`enter.html`/`enter_week.html` — `holePts`/`dnet` stays differential, the `gpts` overall-point block reverted to `net[]`). The 3 duplicates that only render per-hole indicators and read `overall_point_won` straight from `match_results` (`_load_completed_scorecard`, `reports.py`, `standings.py`) needed no change — they never computed their own overall figure.

**Scoring Debug page (`debug/scoring.html`):** per-hole cells show diff-net bold with absolute net in muted parens (diff net drives that Result column). The **total row at the bottom flips emphasis** — absolute net bold with diff net in parens — because that's the number the Overall point box directly below actually uses. A legend above the table spells out which number drives which part.

**Stableford is unaffected** by any of this — each player scores against par independently there; there's no opponent-dependent stroke allocation to get wrong.

### The three recalc tools — don't conflate them

| Tool | Route | Recomputes | Leaves alone |
|---|---|---|---|
| **Recalculate Handicaps** ("↺ Recalc Handicaps") | `POST /handicap/recalc/<season_id>` | Each player's current handicap *index*, one season, from their latest rounds | Match results / points — not touched |
| **Recalc Points** ("🔁 Recalc Points", Admin → Season panel) | `POST /admin/season/<id>/recalc-points` | Hole win/loss + overall point + total for every completed round in *that season*, using each round's already-stored `handicap_at_time_of_play` | Handicaps — never recomputed, explicitly reuses the stored value (`use_existing_hcp=True`) |
| **Rebuild Handicap Timeline** (Admin → More Tools) | `GET/POST /handicap/rebuild` | Everything — handicaps AND net scores AND match points, chronologically, across *every season* in the league | Nothing — full engine, see "Handicap System — Chronological Rebuild Engine" above |

When only match-scoring logic changed (not handicap computation, e.g. this differential-net fix) — **Recalc Points** is the precise, season-scoped, faster tool. Rebuild Handicap Timeline is a safe superset (it'll also fix it, just by doing more work than necessary). Recalculate Handicaps alone won't fix a match-scoring bug — it doesn't touch `match_results` at all.

## Pre-Eligibility Temp Handicap

*(Added 2026-07-02, WP3.15)*

Before a player reaches `min_rounds_for_handicap` (2.04) real rounds, they have no `handicap_history` row, so `get_player_handicap()` falls back to `players.starting_handicap` (commonly `0` — scratch). Fixed by having `rebuild_player_handicap_timeline()` (handicap.py) compute a **temp handicap** for each pre-eligibility round: that round's own gross-vs-par differential × `temp_handicap_percent_member` or `_sub` (2.10/2.11, keyed off that round's own `scorecards.is_sub` flag — a per-round flag, not a permanent player attribute), run through `calc_playing_handicap()` verbatim (same rounding/capping as everywhere else). **Independent per round** — never averaged with other pre-eligibility rounds, never carried into the next round's `entering` value. Marked in `handicap_history.override_reason` via the `PRE_ELIGIBILITY_MARKER_PREFIX` constant (handicap.py) — the single source of truth every other piece of code checks to distinguish "real" from "provisional."

**The double-percent trap, again:** a temp row's `handicap_index` is already a *final* playing-handicap-equivalent value (percent baked in), not a raw index. Any code that fetches "latest handicap_history row" and then runs it through `calc_playing_handicap()` again will silently double-discount it — the same bug class as the WP3.12 double-percent fix, just triggered by a new data shape instead of a code bug. `_recalc_single_round()` avoids this via a **separate** `temp_ph_lookup` parameter that bypasses the normal `handicap_lookup` → `calc_playing_handicap()` path entirely. When adding a new "current handicap" display or calc, always check `override_reason` for the `PRE_ELIGIBILITY_MARKER_PREFIX` before re-applying `calc_playing_handicap()` — six call sites already had this exact bug latent in them (my_stats.py, players.py's Formula Breakdown page, api.py's mobile twin) before it was caught and fixed this session, purely because they pre-dated the temp-handicap feature and had never needed to think about it.

**Display convention:** "current handicap" surfaces across the app (standings, schedule, profile, dashboard, etc.) show the provisional value rather than suppressing it, marked with a small asterisk (`.hcp-provisional-mark` CSS class, `main.css`) + tooltip. Exceptions: CSV exports use a plain-text `(provisional)` suffix instead of HTML; the Records page's "Lowest handicap ever" leaderboard excludes provisional rows entirely (a one-off non-averaged value isn't a real achievement to rank).

**Provisional marker in score entry (2026-07-02):** the pre-eligibility indicator described above is *forward-looking* — it explains that a round *about to be* scored will use a temp handicap, and is only shown while the matchup isn't yet completed. Once completed, `enter.html`/`enter_week.html` swap in the round's actual stored `handicap_at_time_of_play`, and separately check `handicap_history` for a `trigger_round_id`+`PRE_ELIGIBILITY_MARKER_PREFIX` match to show a small clickable `*` (`.hcp-provisional-mark` reused as a `<button>`, same click-toggle tooltip pattern as the eligibility indicator) explaining retrospectively that the round *was* scored provisionally. Two separate indicators, two separate code paths, deliberately — the eligibility button is gated to non-completed matchups and the asterisk only makes sense once a round is done.

**Eligibility-crossing round (2026-07-03, superseding an earlier `d8da205`-era fix):** the "crossing round" is the single round where a player's pool of qualifying differentials first reaches `min_rounds_for_handicap` — i.e. the round whose own score is what makes them eligible. `d8da205` originally gave this round a self-only temp playing handicap (identical formula to the pre-eligibility branch: that round's own differential × member/sub percent), out of a mistaken belief that letting it use its own freshly-averaged index would be a circular "peeking" bug. On review this was wrong on two counts: (1) handicap **index** is computed purely from gross-vs-par differentials — it never depends on playing handicap, strokes, or net scores — so there is no mathematical circularity in a round using its own just-computed index as its own entering handicap; (2) pre-eligibility rounds already establish the identical precedent (each uses its own round's differential to set its own temp handicap), so extending that one round further isn't a new kind of self-reference.

Simplified accordingly: `rebuild_player_handicap_timeline()` (`handicap.py`) no longer special-cases the crossing round. `entering_by_round[round_id]` is now populated with a snapshot (`entering_before_this_round`) taken *before* that round's own eligibility/index computation for every round **except** the crossing round itself, which uses the value just resolved for that round (`entering`, i.e. its own real average, or an anchor override if one exists) — there is no valid prior "real" entering value for it (every round before it was pre-eligibility, which never updates `entering`). `temp_ph_by_round` is populated only by genuinely pre-eligibility rounds, same as always. No separate self-only-temp mechanism, no second `handicap_history` marker value — the crossing round's stored index is real and unmarked, and its playing handicap now flows through the normal `calc_playing_handicap(raw_index, handicap_percent, max_handicap)` path in `_recalc_single_round`, same as every later round. (The `2026-07-03-crossing-round-marker.md` audit doc, which proposed persisting a display marker for the old divergent-Playing-Hcp behavior, is now obsolete — there's no divergence left to mark.)

## Handicap History — Manual Override Transparency

*(Added 2026-07-02)*

Overriding a `handicap_history` row via `override_handicap()` (handicap.py) used to overwrite `handicap_index`/`override_reason` in place — the calculated or provisional value an override replaced was permanently lost. Fixed with two new columns, `pre_override_index`/`pre_override_reason`, populated **only on the first override** of a row (checked via `hh['is_manual_override']` before the UPDATE — if already `1`, only the override itself is updated, the snapshot columns are left untouched). This guarantees the snapshot always reflects the true original calculated/provisional figure, never an intermediate override value from a prior edit. `handicap/player_history.html` shows "Calculated (provisional): X → overridden" under the "Edited" badge when a snapshot exists.

Also fixed the same day: `player_history()`'s table was built by walking `rounds_played` and matching each round to a `handicap_history` row via `trigger_round_id` (or a same-date fallback) — any row with **no** `trigger_round_id` (a standalone override, or a "Recalculate Handicaps" run on a day the player didn't play) was invisible on the page unless its date happened to coincide with an actual played round. Unmatched rows are now merged into the list as their own roundless entries.

## Known Constraints / Gotchas

- **Supabase port:** Use connection pooler port `6543`, not direct `5432` — direct host is IPv6-only and fails with "Network is unreachable" on Render
- **Postgres GROUP BY strictness:** SQLite allows selecting non-aggregate columns not in GROUP BY; Postgres does not. Every non-aggregate SELECT column must appear in GROUP BY. Large source of migration bugs — see `migration.md` Phase 5.3 for the full list of files fixed.
- **HAVING aliases:** Postgres `HAVING` cannot reference SELECT-list aliases. Replace `HAVING alias >= N` with the actual aggregate expression.
- **Dynamic `?` placeholders:** The tokenize-based `?`→`%s` script missed placeholders built via string concatenation (`query += " AND x = ?"`). Three files had these: `handicap.py`, `players.py`, `stats.py` — all fixed.
- **`ensure_tables()` in admin.py:** SQLite-only DDL; guarded with `DATABASE_URL` check — no-ops under Postgres. Tables come from `schema_postgres.sql`.
- **Render free tier spins down** after inactivity — upgrade to $7/mo paid before sharing demo URL publicly.
- **Rapid successive pushes to `main` can produce a spurious "failed"/timed-out deploy** for an intermediate commit that Render supersedes with the next push before it finishes booting — the log shows a clean boot, one successfully-served request, then a graceful `TERM` (not a crash), followed by a port-scan timeout minutes later. Before concluding a deploy actually broke something, check the status of the *latest* commit on `main`, not the one named in the failure notification (2026-07-03).
- **Mixed gross-stats/points queries and `is_absent`:** see "Absence / Ghost-Score Handling" above — never blanket-filter `is_absent` out of a query that also aggregates `match_results.total_points`.
- **`division` vs `division_name` column:** `schema_postgres.sql` uses `division_name`; old queries selected `division` → `UndefinedColumn`. Fixed in `league_info.py`.
- **New columns need THREE things, not one — this caused a production 500 twice in one session:** (1) the column in `schema_postgres.sql`'s `CREATE TABLE` (and the SQLite `init_db.py` equivalent, for local dev), (2) a migration file in `app/migrations/` with `ADD COLUMN IF NOT EXISTS`, **and** (3) that filename actually added to the `additive` list inside `init_db.py`'s `_apply_additive_migrations_postgres()` — writing the migration file alone does nothing, since only files in that list ever get executed against the live Postgres DB. `add_temp_handicap_percent.sql` was written but never added to the list, so it silently never ran, and `_get_settings()`'s defensive-merge loop still crashed with a raw `KeyError` on the missing column (it only guarded against `NULL` values, not columns absent from the row entirely — same root defensive gap in `_PgRow.__getitem__`, worth checking any similar "merge row over defaults" loop for). Separately, `handicap_history`'s override columns (`trigger_round_id`, `is_manual_override`, `override_reason`, `override_by_user_id`, `override_at`) turned out to have **never** been in the schema or a migration at all — they only existed in production via an undocumented direct DB edit, predating migration files being used consistently in this project. Both fixed 2026-07-02; check all three locations whenever adding a column to an existing table. A third instance surfaced 2026-07-03: `matchups.week_label`/`makeup_for_week` existed only via the standalone `migrate_rain_outs.py` script (none of the three locations), 500-ing `admin.panel()` on any fresh DB — fixed the same way (`add_rain_out_columns.sql`).
