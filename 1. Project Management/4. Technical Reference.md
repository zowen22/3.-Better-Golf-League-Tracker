## Architecture

Multi-tenant Flask app. Each league is fully isolated by `league_id`. No data crosses league boundaries.

```
app/
  app.py              # Flask app factory, blueprint registration, context processors
  config.py           # DATABASE_URL, SECRET_KEY env var loading
  database.py         # get_db() — branches SQLite vs Postgres; _PgWrapper + _PgCursorWrapper
  init_db.py          # DB init — Postgres: runs schema_postgres.sql; SQLite: run_migrations()
  schema_postgres.sql # Full 51-table Postgres schema (source of truth for Postgres)
  routes/             # One blueprint per feature: admin, auth, players, scores, standings, etc.
  templates/          # Jinja2 templates mirroring routes structure
  static/css/main.css
  static/js/main.js
```

Legacy SQLite migration scripts (`migrate_*.py`) live in project root — not used under Postgres.

## Conventions

- All queries use `%s` placeholders (psycopg2), never `?` (sqlite3)
- Row access: `row['column_name']` via RealDictCursor — never positional index
- INSERTs that need the new PK use `RETURNING <pk>` — never `cursor.lastrowid` (returns None in psycopg2)
- Dialect branching where needed: `database.is_postgres()` guards SQLite-only code paths
- `enumerate` registered as both Jinja2 global and filter in app.py
- `login_required` / `admin_required` decorators in auth.py — all protected routes use these

## UX Design Patterns

### Score Entry — Input Advance Philosophy

Score inputs follow **row-major (player-centric) advance order**: all holes for player 1, then all holes for player 2, etc. This matches the natural DOM tab order (inputs are laid out as player rows × hole columns in the HTML).

**Rules:**
- Single digit 1–9 → auto-advance to the next input (same player, next hole). Uses the `input` event, not `keydown` — mobile numeric keyboards (iOS/Android) do not reliably fire `keydown` with `e.key` values.
- `Enter` key → same advance.
- Typing `0` or any non-single-digit value (e.g. "10") → stays in cell. Admin enables "Allow double-digit" for scores ≥ 10.
- Advance order is built in JS as a flat array (`_scoreAdvanceOrder`) from `players → holes` loop nesting (not `holes → players`).

**Mobile layout:** Single scrollable table on all screen sizes. The old hole-by-hole card view (with Next/Prev buttons) is retired. Table scrolls horizontally; inputs are 36px wide with `font-size: 16px` to prevent iOS auto-zoom.

**Input type:** Score inputs use `type="text" inputmode="numeric" pattern="[0-9]*"` — NOT `type="number"`. On iOS, `type="number"` can return `this.value = ""` during the `input` event before the browser commits the value, which breaks the auto-advance regex check. `type="text"` with `inputmode="numeric"` shows the numeric keyboard on iOS/Android and gives reliable `this.value` on every keystroke. Range validation (1–20) is enforced server-side.

**iOS focus — confirmed working approach:** `input` event + `requestAnimationFrame(() => next.focus())`. Both sync and rAF calls are made (sync for desktop/Android speed; rAF for iOS). Confirmed working on iOS Safari and Chrome iOS as of 2026-06-13.

**What was tried (in order):**
1. `keydown` + `e.key` — iOS virtual keyboards don't fire reliable `e.key` values
2. `input` + `type="number"` — `this.value` can be `""` during the event on iOS before browser commits
3. `input` + `type="text" inputmode="numeric"` — appeared broken; root cause was actually a JS syntax error (see below)
4. `input` + `type="text"` + `requestAnimationFrame(focus)` — **confirmed working** ✓

**Root cause of all iOS failures:** A missing `</script>` tag before `{% endif %}` in `enter.html` caused a JS syntax error in the holes script block. All holes JS (live calc, stroke dots, auto-advance) was silently dead. The `<script>` for `toggleAbsenceRow` appeared literally inside the unclosed holes `<script>`, and `<script>` is not valid JS syntax. Added `</script>` before `{% endif %}` in commit `37c8fb5`. The iOS focus restriction research was a red herring — `input` + rAF focus works fine once the script executes.

**Why player-centric, not hole-centric:**
Walking a golf hole, each player writes down their own score before moving to the next hole. A player-centric tab order (P1: H1→H2→H3…H9, then P2: H1→H2…) matches how a scorer mentally fills the card — one player at a time. Hole-centric order (H1: P1→P2→P3→P4, then H2…) adds 17+ unnecessary focus jumps across rows, which feels wrong even on desktop and becomes expensive on mobile.

**Related audit item:** P0-5 in `Audits/2026-06-12-score-entry-audit.md`

## Tee Deduplication

The `tees` table stores one row per (color × nine × gender). A course with Gold tees can have up to 6 rows: Gold/Front/M, Gold/Back/M, Gold/Full/M, Gold/Front/W, etc. Without dedup, all pickers show 6 separate entries.

**Key columns:** `tee_color` (bare color e.g. "Gold"; may be NULL — fall back to `tee_name`), `nine` ("front"/"back"/"full"), `gender` ("M"/"W", default "M").

**Web canonical logic** (`scores.py` ~line 265, used in score entry per-player tee dropdown):
1. Fetch all tees for the course ordered by `gender, tee_name, nine`
2. Determine `selected_nine` from the matchup's assigned `tee_id`
3. Filter to tees where `nine == selected_nine`
4. Group by color (`tee_color` or `tee_name` fallback); prefer M representative over W per color
5. Result: one entry per unique color, tee_id points to M rep for that nine

**API shape for iOS** (`GET /api/v1/courses`, implemented 2026-06-20):
```json
{
  "course_id": 1, "course_name": "Pebble Beach",
  "tee_colors": [
    {
      "color": "Gold",
      "tees": {
        "front": {"tee_id": 10, "tee_name": "Gold (Front)", "holes": [...]},
        "back":  {"tee_id": 11, "tee_name": "Gold (Back)",  "holes": [...]},
        "full":  {"tee_id": 12, "tee_name": "Gold (Full 18)", "holes": [...]}
      }
    }
  ]
}
```

**iOS resolution flow:** Schedule returns `tee_nine` per matchup. Tee picker shows color names only. On submit: `tee_colors[selected_color].tees[matchup.tee_nine].tee_id`.

**Same-color, W-only tees** (no M counterpart) get their own color entry — not suppressed.

## Known Constraints / Gotchas

- **Supabase port:** Use connection pooler port `6543`, not direct `5432` — direct host is IPv6-only and fails with "Network is unreachable" on Render
- **Postgres GROUP BY strictness:** SQLite allows selecting non-aggregate columns not in GROUP BY; Postgres does not. Every non-aggregate SELECT column must appear in GROUP BY. Large source of migration bugs — see `migration.md` Phase 5.3 for the full list of files fixed.
- **HAVING aliases:** Postgres `HAVING` cannot reference SELECT-list aliases. Replace `HAVING alias >= N` with the actual aggregate expression.
- **Dynamic `?` placeholders:** The tokenize-based `?`→`%s` script missed placeholders built via string concatenation (`query += " AND x = ?"`). Three files had these: `handicap.py`, `players.py`, `stats.py` — all fixed.
- **`ensure_tables()` in admin.py:** SQLite-only DDL; guarded with `DATABASE_URL` check — no-ops under Postgres. Tables come from `schema_postgres.sql`.
- **Render free tier spins down** after inactivity — upgrade to $7/mo paid before sharing demo URL publicly.
- **`division` vs `division_name` column:** `schema_postgres.sql` uses `division_name`; old queries selected `division` → `UndefinedColumn`. Fixed in `league_info.py`.
