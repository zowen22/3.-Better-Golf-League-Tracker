## Access Notes

*(Established 2026-07-04, once network egress to golfleaguetracker.com was opened)*

- **Use `curl` with a realistic browser `User-Agent`, not the `WebFetch` tool.** GLT's own bot detection returns a `403` to `WebFetch`'s request fingerprint specifically — this is a site-side block, not a sandbox network-policy issue (confirmed: proxy status showed zero relay failures; a plain `curl -A "Mozilla/5.0 ..."` succeeds immediately). Example: `curl -sS -A "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36" "<url>"`.
- **Two canonical starting points** for future exploration:
  - How-To article index: `https://www.golfleaguetracker.com/glthome/help/how-to/`
  - Master settings reference (every configurable GLT setting, categorized): `https://www.golfleaguetracker.com/glthome/about/settingslist`
- **URL scheme is inconsistent on GLT's own site** — most how-to articles live at `/glthome/cms/help/how-to/<slug>`, but a handful use `/glthome/cms/how-to/<slug>` (missing the `help` segment: `adding-subs`, `sub-flighting`, `scoring-against-the-course`, `course/hole-handicaps`). Not a typo on our end — GLT's own inconsistency, noted so no one "fixes" it by mistake.
- **Authenticated login (for pages behind GLT's login wall, e.g. all of Part 4's Rules/Stats pages) — confirmed working 2026-07-09.** GLT uses a standard ASP.NET login form with an anti-forgery token. **Credentials (@user-authorized to store here, 2026-07-09 — low security risk, this is a test/demo league account): League ID `Buckeye`, password `skypilot`.** Steps, all via `curl` with the same realistic `User-Agent` as above, using one cookie jar throughout:
  1. **GET the target page first** (any authenticated URL, e.g. `/glthome/statistics/`) with `-c <cookiejar>` — it 302-redirects to `/glthome/security/login?fp=<original-path>`; let curl follow (`-sSL`) so you land on the actual login page and its Set-Cookie (anti-forgery cookie) lands in the jar.
  2. **Scrape the anti-forgery token** out of that page's HTML: `<input name="__RequestVerificationToken" type="hidden" value="...">`. It's a long opaque string, unique per page load — must be freshly scraped each session, can't be reused/hardcoded.
  3. **POST to `/glthome/security/loginaction`** with `-b <cookiejar> -c <cookiejar>` (send the anti-forgery cookie back, capture the new session cookie) and form fields: `__RequestVerificationToken` (scraped value), `UserId` (League ID), `Password`, `fp` (the original redirect-target path from step 1, e.g. `/glthome/statistics/`), `ddYearNum=2026`, `RememberUserId=false`, `token=` (empty). Follow redirects (`-sSL`) — a successful login lands you directly on the originally-requested page, already rendered.
  4. **Reuse the same cookie jar (`-b <cookiejar>`) for every subsequent authenticated request** in the session — no need to re-login per page.
  - One-shot copy/paste template, credentials already filled in — just change `TARGET` to whatever page you actually want:
    ```bash
    UA="Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36"
    COOKIEJAR=/tmp/glt_cookies.txt
    TARGET="https://www.golfleaguetracker.com/glthome/statistics/"

    curl -sSL -A "$UA" -c "$COOKIEJAR" "$TARGET" -o /tmp/glt_login_page.html
    TOKEN=$(grep -o '__RequestVerificationToken" type="hidden" value="[^"]*"' /tmp/glt_login_page.html | sed 's/.*value="//;s/"$//')

    curl -sSL -A "$UA" -b "$COOKIEJAR" -c "$COOKIEJAR" \
      -e "https://www.golfleaguetracker.com/glthome/security/login?fp=/glthome/statistics/" \
      --data-urlencode "__RequestVerificationToken=$TOKEN" \
      --data-urlencode "ddYearNum=2026" \
      --data-urlencode "fp=/glthome/statistics/" \
      --data-urlencode "UserId=Buckeye" \
      --data-urlencode "Password=skypilot" \
      --data-urlencode "RememberUserId=false" \
      --data-urlencode "token=" \
      "https://www.golfleaguetracker.com/glthome/security/loginaction" \
      -o /tmp/glt_page.html

    # then for any further page: curl -sSL -A "$UA" -b "$COOKIEJAR" "<url>" -o /tmp/out.html
    ```
  - Confirmed via this exact method: full access to `/glthome/statistics/` (~28 report pages) and `/glthome/league/leaguerules` — no bot-blocking encountered once authenticated, same as the public pages above.
- The settings-list page is large (209 settings across 24 categories — verified exact count, see below) and best processed one category at a time rather than in one pass — see the Settings Reference table below for what's been reviewed vs. not.
- **GLT's own category filter dropdown is out of sync with the page**: it lists only 19 of the 24 categories (missing Administrative, Best Ball Points, Beta Testers, Stroke Play Points - Team, Team Totals Points) — those 5 are only visible via the default "All Categories" view. Confirmed via two independent counts (`id='table-CategoryName'` elements and `class="tableHeader"` divs both = 24) — not a scraping miss on our end, a real inconsistency on GLT's site.

### Tier 1 — Fix regardless of parity planning (real defects, not gaps)

1. **Stableford scoring mode never activates** — `_settings_scoring_mode()` (`app/routes/scores.py:154-162`) reads the wrong DB column (`scoring_type` instead of `scoring_mode`). Selecting "Stableford" in league settings has zero effect; every round scores as match play. One-line column-name fix. Not a design decision — just broken. **FIXED 2026-07-06 (`be845e6`).**

### Tier 2 — Highest-leverage closure work (one mechanism fixes many settings at once)

2. **Absent Players/Subs point-clamp gap** — 10 of 12 GLT settings in this category are variations on one missing capability: a per-category (player/team) × (min/max) × (vs-course/vs-sub) point-clamping grid. BGLT has only a single 3-way `absence_overall_point_policy` switch that touches the overall point only. A general point-clamp layer would close 10 settings in one project. The 11th (Absent Player Penalty Strokes) is a structural mismatch, not a clamp — see Tier 5.
3. **`ab_designation_method` dead setting** — most consequential of the 5 dead settings found this audit. A/B role assignment (`scores.py:573-577`) always sorts by current playing handicap every round; the setting that's supposed to control this (`'weekly'`/`'season'`) is saved, displayed, and cloned by the season wizard but **never read**. Determines match pairings and points every week, for every league. Also directly corresponds to GLT's "Static Flights" setting (General Information category) — same gap, two names.
4. **Sub-request email notification system** — 5 of E-mails' 10 settings (Copy League Officers, Copy Teammate, Sub Request Message, Reminder Days, Verification Required) all depend on one missing prerequisite: BGLT has no sub-request email notifications at all. Building that one feature would close 5 settings at once, similar in shape to item 2 above but smaller.

### Tier 3 — Cheap, well-scoped wins (small settings-branch additions onto logic that already exists)

5. **4 remaining dead settings** — wiring each up is a narrow, mechanical fix: `diff_calculation_type='whs'` (handicap.py always computes par-based differentials — this one also has a real design decision attached, see Tier 5), `max_score_over_handicap` (never read), `match_play_points_per_hole`/`match_play_overall_point` (hardcoded 2.0/0.0/1.0 in `calc_match_play()`), `skins_config.handicap_percent` (never read in `skins.py`). **1 of 4 done: `max_score_over_handicap` FIXED 2026-07-06 — see Work Packages.**
6. **Hole Tie Calculation / Match Tie Calculation** — BGLT hardcodes ties to always split points; GLT offers an "award nothing" alternative. Small toggle onto existing logic.
7. **Match Play Point Method** — whether hole points count toward the total or are bragging-rights-only. BGLT's current behavior is already one of GLT's two options; just needs the toggle.
8. **Schedule: Show Subs on Schedule / Show Tee Times** — both narrow settings-branch additions onto template logic that's already ~90% there (schedule page has no sub-display path at all; tee-time visibility is auto-detected rather than an explicit toggle).
9. **E-mails: Blind Copy (BCC) / E-mail Signature / Allow Players to Send E-mails** — 3 standalone small additions, independent of the sub-request-notification feature above.

### Tier 4 — Real but lower-stakes (admin/UX conveniences, not weekly-impact items)

10. Bonus Points for Showing Up (regular + sub) — no attendance-bonus concept exists.
11. Swap A/B Matchups — no manual per-week pairing override exists.
12. Skins: per-par/round-total stroke caps, half-strokes, multipliers, worst-score-eligibility cutoff, gross/net-simultaneous — real gaps, but narrow and specific to the skins side-pot, not standings-affecting.
13. **12 narrow General/Administrative/Scorecards/Signup gaps** — hide-player-detail, phone auto-formatting, time zone, custom theme, combine-teebox-info, round notes, distance label, pace-of-play time, scorecard message, days-between-rounds, league playfield (course vs. simulator), minimum tee times. Each independent and small; no shared mechanism to batch-fix, unlike Tier 2/3 items above.
14. **Standings: Combine Same Dates, Number of Scheduled Rounds Shown** — real gaps, narrow scope (a same-day-round merge display rule, and an "upcoming rounds" section).
15. **Standings: Show Total Points for Segment** — partial match; `segment_pts` is already computed but only surfaced on the per-player scorecards detail view, not the main League Standings page GLT's setting describes. Cheaper than a full gap since the data already exists — likely just a template/display fix.

### Tier 5 — Bigger lifts (touch the core scoring/handicap/standings loop, not just a settings branch)

16. **Stableford's 9-tier point curve** vs. BGLT's 5 hardcoded values, plus front9/back9 point splits (both Match Play and Stableford) — real gaps, but changes to the scoring engine itself, not a settings branch.
17. **WHS-style slope/rating handicap differential** (`diff_calculation_type='whs'`) — GLT's own recommended default leans this way; this is a real product decision (par-based vs. full WHS math), not just a dead-setting wiring fix. Flagged here because it's bigger than the other dead settings despite being technically one of the 5.
18. **Absent Player Penalty Strokes** — structurally different ghost-scoring model (GLT: separately-tunable "course plays for them" pseudo-player; BGLT: ghost-scores using the absent player's own handicap). Not a missing parameter on the same mechanism — would require rethinking the ghost-scoring approach itself.
19. **Standings Sort Method / Sort Order / Force Decimal Places** — all three would touch the core standings query/rendering path (currently a hardcoded `ORDER BY total_pts DESC` and default Jinja number formatting), not just add a settings branch.
20. **Multi-segment support (up to 4 segments)** — BGLT supports exactly one segment window; this is 4 of League Size and Length's 6 settings, all the same underlying gap.

### Tier 6 — Out of scope for "gap closure" (new product surface, not a fix)

21. Stroke Play (individual/team/vs-field), Best Ball, Team Totals, Low Net as scoring formats — BGLT has exactly one 2-player-team match-play/stableford structure; adding any of these is a second scoring engine, not a settings tweak.
22. Team Skins, League Groups, calculated-handicap-vs-course-handicap distinction, individual/scramble league types (BGLT is hardcoded 2-player-team) — no prerequisite concept exists in BGLT's schema at all.

### Not tiered — reviewed, no gap found (workflow comparison, not settings)

23. **Season rollover workflow** ("Starting a New Season" how-to vs. WP3.20's "Start Another Season" wizard) — full step-by-step comparison found **no new functional gaps**; every GLT step is either matched, not applicable (constraint doesn't exist in BGLT), or a deliberate design difference. One item genuinely needs @user's judgment rather than either Fable planning or silent adoption: **GLT requires a 30-day cooldown after a season ends before offering renewal; BGLT offers it immediately.** Not clearly a bug or a gap — just worth confirming intentional. See "Season Rollover Workflow" in Part 2 (Workflow Parity) below for the full step-by-step.

## Full Settings Gap Audit (2026-07-04)

*Audit against the remaining categories of `/glthome/about/settingslist`, done via two Explore agents cross-checking BGLT's actual code (not just the settings schema) — full citations in each row. Purpose: identify gaps for @user + Fable to plan closure work from; no fixes made here.*

### ⚠️ Live bug found during this audit (not a gap — a real defect)

**Stableford scoring mode silently never activates.** `_settings_scoring_mode()` (`app/routes/scores.py:154-162`) reads `settings['scoring_type']` — the **net/gross** toggle (`schema_postgres.sql:109`, values `'net'`/`'gross'`) — when it should read `settings['scoring_mode']`, the actual **match_play/stableford** format toggle (`schema_postgres.sql:140`). Since `scoring_type` never equals `'stableford'`, every `if scoring_mode == 'stableford':` check in the scoring engine (`scores.py:1027,1351,2139,2145`) is permanently unreachable — an admin selecting "Stableford" in league settings has **zero effect on live scoring**; every round scores as match play regardless. Root cause: `_settings_scoring_mode`'s docstring literally says *"Column is 'scoring_type' in the DB"* — a leftover from an older fix (`1e0e07c`, "Fix KeyError: 'scoring_mode' — DB column is scoring_type", predates the `scoring_mode` column existing at all, added later same week). This reads as a genuine, standalone bug fix, not a feature-parity decision — flagged separately in Work Packages as higher priority than the gap items below. **FIXED 2026-07-06 (`be845e6`).**

2. **Absent Players/Subs point-clamp gap (10 of 12 settings)** is the single highest-leverage closure item found in this pass — one general min/max point-clamp mechanism (player/team × min/max, applied regardless of opponent type) would close most of a whole category at once, unlike Skins where every gap requires a separate, narrow change to `_calculate_skins`.
3. **Skins gross/net exclusivity** (can't run both pots in the same round) is a real but lower-urgency partial match — most leagues run one or the other, not both.
4. **Skins per-par/half-stroke/round-total stroke caps and multipliers** are genuine gaps but narrow, cosmetic-adjacent features (skins-specific stroke tuning) — lower priority than the absence point-clamp work above.
5. **Absent Player Penalty Strokes** (structural ghost-scoring mismatch) and the **Skins Not-applicable cluster** (team-skins, league-groups, calculated-handicap distinction) all require a prerequisite feature BGLT doesn't have — flag for the same "new product surface" conversation as stroke-play/best-ball/team-totals, not bundled into ordinary gap-closure work.

## Season Rollover Workflow (How-To Comparison vs. WP3.20)

*(Reviewed 2026-07-04 — a workflow-level comparison, not a settings comparison: does GLT's actual season-rollover *process* match what BGLT's "Start Another Season" wizard does? Source: [Starting a New Season](https://www.golfleaguetracker.com/glthome/cms/help/how-to/renewal), full text read. "League Year-End Maintenance" turned out to be just a link index to 3 other articles — League Playoffs, Year-End Outing, and this one — not itself substantive, so no separate comparison needed for it.)*

**GLT's process** (verbatim structure): once a season's last scheduled date is **30+ days old**, the admin gets prompted at login to "copy" the league to a new year. The copy automatically brings over players, courses, tee times, and settings. The admin then walks a checklist of League Setup steps for the new season: update "Number of Players" (setting #69) if roster size changed; add new players / mark departing players as either a **sub** or **inactive** (inactive = admin-only visible, "equivalent to deleting"); reset tee times if changed; set the new league start date; **re-assign player/team numbers** if the roster or teammates changed (used by GLT's scheduling algorithm); generate the new schedule (auto-generator or manual).

**BGLT's wizard** (`app/routes/seasons.py:125-` `start_next()`), checked directly against each step above:

| GLT Step | BGLT Equivalent | Verdict |
|---|---|---|
| 30-day-post-season wait before renewal is offered | `season_is_over()` (`seasons.py:38-59`) offers the CTA as soon as the last matchup date (or `end_date`) is in the past — no cooldown period | **Different, deliberately not a gap to close** — BGLT is more permissive (available immediately). Worth understanding *why* GLT waits 30 days before assuming BGLT's earlier availability is strictly better — could be tied to GLT's own billing/subscription-renewal cycle (a paid-service administrative reason), not a workflow best practice. Flag for @user judgment, not an automatic fix. |
| Players, courses, tee times copied automatically | BGLT: players and courses are league-scoped already (not season-scoped), so there's nothing to "copy" — they're just already there. Matches the *outcome* GLT describes, via a different (arguably simpler) mechanism. | **Match** (different mechanism, same result). |
| Settings copied automatically | BGLT: **optional** via `bring_settings` checkbox, not automatic | **Partial** — BGLT gives the admin a choice GLT doesn't; defaults aside, this is arguably a deliberate improvement (a league changing formats mid-renewal can start clean) rather than a gap, but it does mean BGLT's wizard requires an active choice where GLT just does it. |
| Teams copied automatically | BGLT: **optional** via `bring_teams` checkbox | **Partial**, same reasoning as Settings above. |
| Update "Number of Players" setting if roster size changed | N/A — BGLT has no "Number of Players" setting/cap at all (confirmed **not applicable** in the League Size and Length deep-dive) | **Not applicable** — nothing to update because the constraint this step manages doesn't exist in BGLT. |
| Add new players / mark departing players as sub or inactive | BGLT: `players.deactivate()`/`reactivate()` routes (`players.py:499-531`), `active` column, confirmed player-facing views filter to `active = 1` | **Match** — same underlying ability (mark a departing player inactive rather than deleting, admin can still see/reactivate them), different route names. |
| Reset tee times if changed | BGLT: tee/course setup is league-scoped, not tied to season rollover as a distinct step; schedule generation assigns tee times per matchup directly | **Different mechanism, no gap identified** — BGLT doesn't need a distinct "reset tee times for the new season" step since tee-time assignment happens at schedule-generation time regardless of season boundary. |
| Set new league start date | BGLT: `start_date`/`end_date` fields on the new season, set directly in the wizard form | **Match**. |
| Re-assign player/team numbers (used by GLT's scheduling algorithm) | BGLT: `generate_round_robin(teams)` (`schedule.py:43`) works directly from the team list — no manual numbering step exists or is needed | **Not applicable** — this GLT step exists to feed their specific scheduling algorithm's numbering requirement; BGLT's scheduler doesn't have that requirement, so there's no equivalent step to be missing. Simplification, not a gap. |
| Generate schedule (auto or manual) | BGLT: `/schedule/generate` route, round-robin auto-generator + manual entry | **Match**. |

**Net for Fable**: this workflow comparison did **not** surface any new functional gaps in WP3.20 — every GLT step either has a direct BGLT equivalent, is not applicable because the underlying constraint doesn't exist in BGLT, or is a deliberate design difference (optional vs. automatic settings/teams carry-over) rather than a missing capability. The one item worth a judgment call rather than silent adoption is the **30-day cooldown before renewal is offered** — not clearly a bug or a gap, but worth @user weighing in on whether BGLT's immediate-availability behavior is intentional or worth adding a similar grace period for (e.g. to avoid an admin accidentally rolling over before playoffs/final stats are settled).
