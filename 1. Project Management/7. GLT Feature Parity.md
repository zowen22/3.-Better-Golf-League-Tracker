## Access Notes

*(Established 2026-07-04, once network egress to golfleaguetracker.com was opened)*

- **Use `curl` with a realistic browser `User-Agent`, not the `WebFetch` tool.** GLT's own bot detection returns a `403` to `WebFetch`'s request fingerprint specifically — this is a site-side block, not a sandbox network-policy issue (confirmed: proxy status showed zero relay failures; a plain `curl -A "Mozilla/5.0 ..."` succeeds immediately). Example: `curl -sS -A "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36" "<url>"`.
- **Two canonical starting points** for future exploration:
  - How-To article index: `https://www.golfleaguetracker.com/glthome/help/how-to/`
  - Master settings reference (every configurable GLT setting, categorized): `https://www.golfleaguetracker.com/glthome/about/settingslist`
- **URL scheme is inconsistent on GLT's own site** — most how-to articles live at `/glthome/cms/help/how-to/<slug>`, but a handful use `/glthome/cms/how-to/<slug>` (missing the `help` segment: `adding-subs`, `sub-flighting`, `scoring-against-the-course`, `course/hole-handicaps`). Not a typo on our end — GLT's own inconsistency, noted so no one "fixes" it by mistake.
- The settings-list page is large (209 settings across 24 categories — verified exact count, see below) and best processed one category at a time rather than in one pass — see the Settings Reference table below for what's been reviewed vs. not.
- **GLT's own category filter dropdown is out of sync with the page**: it lists only 19 of the 24 categories (missing Administrative, Best Ball Points, Beta Testers, Stroke Play Points - Team, Team Totals Points) — those 5 are only visible via the default "All Categories" view. Confirmed via two independent counts (`id='table-CategoryName'` elements and `class="tableHeader"` divs both = 24) — not a scraping miss on our end, a real inconsistency on GLT's site.

## Handicap Settings / Calculation

*(Reviewed 2026-07-04, once egress was opened — sources: [Recommended Handicap Settings](https://www.golfleaguetracker.com/glthome/cms/help/how-to/common-handicap-options), [Limiting Strokes per Hole for Handicapping](https://www.golfleaguetracker.com/glthome/cms/help/how-to/strokes-per-hole-limit))*

| GLT Setting / Option | BGLT Equivalent | Notes |
|---|---|---|
| Setting #70, "Number of Rounds Required To Establish a Handicap" (GLT recommends 2) | `league_settings.min_rounds_for_handicap` (default `2`) | Exact match on the recommended default. |
| Setting #71, "Number of Most Recent Rounds to Calculate Handicap" (GLT recommends 6) | `league_settings.rounds_to_average` (default `4`) | BGLT's default window is narrower (4 vs. GLT's recommended 6) — both are admin-configurable, not a capability gap, just a different starting default. |
| Setting #40, "Number of High Rounds to Throw Out" (GLT recommends `-33`, i.e. **33% of scores**, not a fixed count) | `league_settings.high_scores_to_drop` (default `1`, a fixed **count** of rounds) | **Real methodology difference.** GLT's high/low-drop settings are percentage-of-pool based (scales with how many rounds exist); BGLT's are a fixed round count regardless of pool size. Worth a product decision: keep BGLT's fixed-count model (simpler, predictable) or add a percentage-based option to match GLT exactly. |
| Setting #53, "Number of Low Rounds to Throw Out" (GLT recommends 0) | `league_settings.low_scores_to_drop` (default `0`) | Same fixed-count-vs-percentage caveat as above; default values match. |
| Setting #38, "Handicap Percent" (GLT recommends `1`, i.e. 100%) | `league_settings.handicap_percent` (default `90.0`) | Both configurable per-league; defaults differ (GLT recommends no reduction, BGLT defaults to the traditional 90% "playing handicap" convention) — not a gap, a different default. |
| Setting #36, "Maximum Calculated Handicap" (GLT recommends `54.0` for 18-hole leagues, `27.0` for 9-hole) | `league_settings.max_handicap_index` (season default `18.0`) + `platform_settings.max_handicap_index_ceiling` (site-wide ceiling, `54.0`) | BGLT's site-wide ceiling already matches GLT's 18-hole recommendation exactly; the per-season default (`18.0`) is far more conservative than GLT's recommended ceiling — admins can already raise it up to the platform ceiling, so this is a default-value difference, not a missing capability. No 9-hole-specific ceiling variant exists in BGLT today (GLT recommends 27.0 for 9-hole; BGLT doesn't distinguish 9 vs 18 for this setting). |
| Setting #112, "Handicap Calculation Type": **"2020 Rules (Custom)"** (course-difficulty/slope-adjusted) vs. **"Par"** (simple par-based, no course-handicap conversion) | `league_settings.diff_calculation_type` — `'par'` is fully implemented; `'whs'` (slope/rating) is a **dead dropdown option** (`admin/settings.html` offers it, but `handicap.py`'s engine always computes par-based differentials regardless of the selection) | **Directly resolves the open WP3.1 backlog question** ("implement real WHS slope/rating math, or remove the dropdown option?") — GLT's own docs confirm course-difficulty/slope-adjusted handicapping is a real, commonly-used mode (their own recommended default, in fact), not a niche USGA-only feature. This is a genuine parity gap, not just dead UI — leans toward "implement it" over "remove the option." |
| "Hole Stroke Limit" section — per-hole max score for handicap purposes, **"Net Double Bogey"** as the common preset, up to **6 different handicap-tiered ranges** (e.g., <10 hcp capped at 7, ≥10 hcp capped at 8), and an **automatic +5 adjustment when a player has no established handicap yet** | `league_settings.max_score_per_hole` (single global int, unset by default) + `max_score_action` (`'warn'`/block) | **Real gap.** BGLT has one global per-hole cap (or none) applied uniformly; GLT supports handicap-tiered ranges (different caps for different skill levels) and a "Net Double Bogey" (par + 2, adjusted for strokes received) preset rather than a raw fixed number. ## Full Settings Gap Audit (2026-07-04)

*Audit against the remaining categories of `/glthome/about/settingslist`, done via two Explore agents cross-checking BGLT's actual code (not just the settings schema) — full citations in each row. Purpose: identify gaps for @user + Fable to plan closure work from; no fixes made here.*

### ⚠️ Live bug found during this audit (not a gap — a real defect)

**Stableford scoring mode silently never activates.** `_settings_scoring_mode()` (`app/routes/scores.py:154-162`) reads `settings['scoring_type']` — the **net/gross** toggle (`schema_postgres.sql:109`, values `'net'`/`'gross'`) — when it should read `settings['scoring_mode']`, the actual **match_play/stableford** format toggle (`schema_postgres.sql:140`). Since `scoring_type` never equals `'stableford'`, every `if scoring_mode == 'stableford':` check in the scoring engine (`scores.py:1027,1351,2139,2145`) is permanently unreachable — an admin selecting "Stableford" in league settings has **zero effect on live scoring**; every round scores as match play regardless. Root cause: `_settings_scoring_mode`'s docstring literally says *"Column is 'scoring_type' in the DB"* — a leftover from an older fix (`1e0e07c`, "Fix KeyError: 'scoring_mode' — DB column is scoring_type", predates the `scoring_mode` column existing at all, added later same week). This reads as a genuine, standalone bug fix, not a feature-parity decision — flagged separately in Work Packages as higher priority than the gap items below.

### Point Rules & Scoring Formats

Reading GLT's full descriptions confirms the granularity gap precisely: GLT allows **9 distinct configurable point values** (double-eagle/albatross through triple-bogey-or-worse, plus separate -4/-5 tiers) versus BGLT's **5 hardcoded values** (`scores.py:133-144`) — meaning even an unusually good hole (better than eagle) or unusually bad one (worse than double bogey) can't be differentiated in BGLT; GLT's settings can. Front9/back9 individual and team Stableford point splits are the same gap as Match Play's. "Stableford Best Ball" (best-of-two-teammates scoring) has no BGLT equivalent (no best-ball concept at all, consistent with the earlier finding).

Ranked by "how much does this affect what a league actually experiences," not by setting count:

2. **`ab_designation_method` being dead** is the most consequential newly-found dead setting — it directly determines who plays whom and who scores which points, every week, for every league.

### Skins (22 settings) — `app/routes/skins.py`, `skins_config`/`round_skins_settings`/`round_skins_participants`/`skins_results` tables

One flat, single-pot, individual-only skins system. Real gaps: no separate gross/net enable flags (one `default_gross_net` toggle, `schema_postgres.sql:442`); no skins label/multiplier/weekly-pot fields; no max-strokes-per-par eligibility cap (`_calculate_skins`, `skins.py:33-99`, uses every hole unconditionally); `skins_config.handicap_percent` (`schema_postgres.sql:443`) exists but is **never read** — net scores come from the league-wide handicap calc, not a skins-specific method/percent (a 4th dead setting, arguably); no team-skins concept at all; no tie logic, league-group scoping, or "worst score that can win" setting. Carry-over exists as a single dollar amount (`round_skins_settings.carried_over_amount`), not gross/net-split.

### Standings (8 settings) — `app/routes/standings.py`

No standings settings section exists at all in `admin/settings.html` (only Scoring/Handicap/Max-Score/Playoffs/Skins/Self-Reporting/Segments/Tiebreakers). Sort is hardcoded (`ORDER BY total_pts DESC`, `standings.py:135,160`). BGLT's **Tiebreakers** feature (head-to-head/points%/all-play/scoring-avg priority, `standings.py:171-341`) is a real but different-mechanism analog to "Standings Sort Method" (tiebreak-only, not a full sort choice). No equivalent for "Combine Same Dates," "Force Decimal Places," "Show Zero Points," or "Number of Scheduled Rounds Shown" (the last is a query param on specific views, not a setting).

### E-mails (10 settings) — scattered across `leagues` table columns + `app/routes/email_config.py` (no unified settings page)

From-address/name implemented and matching (`smtp_from_email`/`smtp_from_name`). Player email opt-out exists (`email_opt_out*` columns) but is a *receive* opt-out, not a *send* permission — different mechanism from "Allow Players to Send E-mails." ### League Size and Length (6 settings)

Holes-per-round matches (`holes_per_round`). Player count is hardcoded to 2 per team (not applicable — BGLT's fixed 2-player-team model doesn't have this as a variable). Segments: BGLT supports exactly **one** segment window (`segment_start_week`/`segment_end_week`); GLT supports up to 4 — real gap on multi-segment granularity, partial equivalent for a single split.

### Absent Players and Subs (12 settings) — `scores.py` ghost-scoring logic, `absence_overall_point_policy`

Materially different mechanism, not just missing settings. BGLT ghost-scores an absent player using **their own playing handicap against par** (no configurable penalty-strokes scratch model like GLT's "0-handicap-minus-penalty" concept). Overall-point handling is one league-wide 3-way policy (`always`/`never`/`excused_only`) — not the 10 granular per-player/per-team numeric min/max point caps and floors GLT exposes (all 10 are real gaps; hole points are computed against the ghost score with no clamping at all, `standings.py:803-835`).
