**Templates:** `admin_list.html` reordered (All Contests table now above the creator form, per @user), Contest Name input removed, "Every week" checkbox added (JS-hides the Week dropdown when checked), delete button branches to a single Delete (non-recurring) or Delete Week▾/Delete Series (recurring) pair. `admin_edit.html` results table expanded to the GLT-matched column set (Week #, Date, Course, Contest Winner, Result, Hole #, Distance, Amount Won, Comments) with inline Edit/Save/Cancel per row (vanilla JS row-swap, no framework); Calculate Results gets a week-picker when the contest is recurring. `season.html` (member view) gets the same redundant-name fix and the same new read-only columns, shown only for recurring contests to avoid cluttering simple one-off contest cards.

## 2026-07-03 — Crossing-Round Handicap Simplification (Superseding `d8da205`)

*Status: `COMPLETED`*
*Type: `Manual`*

### What Was Done
User asked what a "crossing round" was; my explanation included two factual errors — claiming a "formula loop" justified the `d8da205` self-only-temp-handicap mechanism for the crossing round, and claiming pre-eligibility rounds "never use their own score." User correctly pushed back on both: handicap index is computed purely from gross-vs-par differentials, never from playing handicap/strokes/net, so there's no circularity in a round using its own freshly-computed index as its own entering handicap; and pre-eligibility rounds do in fact use their own round's differential to set their own temp handicap (`rebuild_player_handicap_timeline()`'s `else` branch), directly contradicting the second claim.

Re-derived the actual (weaker) justification for `d8da205`'s original design — a golf-handicapping fairness convention ("know your handicap before you play"), not a math constraint — and presented two options: keep `d8da205`'s mechanism, or simplify by letting the crossing round use its own real average directly. User chose simplify.

**Implementation** (`app/routes/handicap.py`, `rebuild_player_handicap_timeline()`):
- Replaced the per-round `entering_by_round[rnd['round_id']] = entering` write at the top of the loop with a local snapshot `entering_before_this_round = entering`, taken before that round's own eligibility/index computation.
- Removed the `if is_crossing_round:` self-only-temp-handicap block that previously lived inside the `elif new_index is not None:` branch — that branch is now just the `handicap_history` INSERT + `entering = new_index`, nothing else.
- Added one line at the end of the per-round loop body: `entering_by_round[rnd['round_id']] = entering if is_crossing_round else entering_before_this_round` — every round uses the prior-rounds-only snapshot except the crossing round, which uses the value just resolved for itself (real average, or an anchor override if present).
- `temp_ph_by_round` is now populated only by the genuine pre-eligibility `else` branch, matching pre-`d8da205` semantics for that dict.
- No changes needed to `_recalc_single_round` or `rebuild_league_handicaps_and_scores` — the crossing round now simply falls through `_recalc_single_round`'s standard `current_handicap()`/`calc_playing_handicap()` path, same as any post-eligibility round.

## 2026-07-02 — Ghost Score Correctness: Edit Scores Fix, Matrix Atomicity, League-wide Stat Audit

*Status: `COMPLETED`*
*Type: `Manual` (planned via ExitPlanMode, then implemented)*

### What Was Done

User reported: Wil & Rosie, both absent (ghost-scored) and both handicap-overridden in Week 5, showed a Net far below par instead of exactly par. Root-caused via direct code reading + one Explore agent trace (three hypotheses checked and ruled out — `_recalc_single_round`/`_process_scores` are both internally consistent) before finding the actual cause in a third path.

**Primary bug — `admin.py` "Edit Scores" tool had no concept of ghost/absent scorecards.** `edit_scores()` pre-filled every scorecard's stored gross — including ghosts — into an editable form input with no visual distinction. `_save_edited_scores()` then recomputed `net_score = gross - strokes_on_hole(CURRENT handicap)` uniformly for every player, using whatever handicap is stored *right now*. If that gross was a ghost value synthesized under an older handicap (before a since-applied override), any Edit Scores save on that round — even one only fixing a teammate's score — silently locked in a corrupted, too-low net. Fixed: `_save_edited_scores` now regenerates a ghost's gross fresh from the current handicap (`par + strokes_on_hole`) and sets net directly to `par` (the established ghost convention used everywhere else), ignoring whatever was submitted/pre-filled. `edit_scores.html` now shows ghost rows as non-editable with a "GHOST" badge instead of presenting inputs whose edits get silently discarded.

**Likely trigger — Handicap Matrix override tool's silent failure mode.** `matrix_update()` committed the `scorecards.handicap_at_time_of_play` override in its own transaction *before* attempting to rebuild/resync scores via `rebuild_league_handicaps_and_scores()`; that call was wrapped in a bare try/except that swallowed any exception into an unused `recalc_errors` field, never rolled back, and the frontend never read `recalc_errors` (only checked `data.ok`, which was always `true`). A failed resync left a round's handicap changed without its ghost `hole_scores` regenerated to match — exactly the precondition Edit Scores then corrupts. Fixed: override + resync now commit together in one transaction (or not at all on failure, with an explicit `db.rollback()` so a connection with an open transaction never returns to the pool), and the admin now sees a visible alert if the resync failed.

**League-wide stat leakage audit** (user's explicit follow-up: "ensure ghost scores don't count toward stats in any way"). Launched 3 parallel Explore agents to audit every player-facing stat computation in `reports.py`, `records.py`, `my_stats.py`, `players.py`, `standings.py`, `api.py`, `schedule.py`, `main.py` for missing `is_absent = 0` filters. ## 2026-06-30 — Chronological Handicap Rebuild Engine

*Status: `COMPLETED`*
*Type: `Manual`*

### What Was Done

**Root cause analysis (two related bugs):**
- Bug 1: editing/resaving a past round's scorecard overwrote that round's own historical handicap with the player's CURRENT handicap, instead of the handicap that was correct as of that round's date.
- Bug 2: the existing "cascade forward" mechanism (meant to propagate a correction into later rounds) was also broken — it flattened every future round to a single "latest" handicap value instead of correctly re-deriving a chronological progression.
- Root cause of both: every recalc path derived handicaps via "latest `handicap_history` row right now" queries, never via true date-ordered replay.

**New chronological-replay engine (`routes/handicap.py`):**
- `rebuild_player_handicap_timeline(db, player_id, league_id)` — Phase 1, per-player. Walks all of a player's completed rounds oldest→newest; for each round, looks up that round's own season's settings (window, drop counts, %, max, carry-across); computes the windowed/dropped/averaged differential pool using only rounds up to and including the current one (correctly date-bounded by construction of the walk, no date-range SQL needed); respects manual-override anchors (`handicap_history.is_manual_override=1` keyed by `trigger_round_id`) by substituting the anchor's value as the entering handicap for the next round; deletes and reinserts all auto-computed rows with `calculated_date` set to the round's actual date. Returns `{round_id: entering_handicap_index}`.
- `rebuild_league_handicaps_and_scores(db, league_id)` — Phase 2, league-wide orchestrator. Runs Phase 1 for every player, merges into a `{round_id: {player_id: raw_index}}` lookup, then walks every completed non-bye matchup chronologically, re-scoring each (net scores, A/B roles, match points) using each player's point-in-time handicap instead of "whatever's latest". Does not commit — caller controls the transaction, which enables an apply-then-rollback preview.

**Wired into the 5 call sites that actually cascade:**
- `_process_scores` (scores.py) — replaces the old two-step "recalc handicap loop" + "`_recalc_future_rounds` cascade" with one call to the new engine.
- `matrix_update`, `clear_scorecard_overrides`, `clear_handicap_override` (handicap.py).
- Settings-save cascade (admin.py) — now also recomputes the underlying index using just-changed settings, not just re-scoring with stale `handicap_history` values.
- Deliberately **not** rewired: `score_import.py`, `self_report.py`, `api.py` (4 sites), `migration.py` — these call the legacy `recalc_handicap_for_player` for a one-off recalc with no cascade, so they don't exhibit the bug. Documented as a known gap in `handicap.py`'s module docstring rather than silently left unflagged.
- Deleted `_recalc_future_rounds` (scores.py) as dead code once its 3 callers were rewired.

**Latent bug fixed as a side effect:**
- `_recalc_single_round`'s playing-handicap resolution only respected `hcp_manually_overridden` when `use_existing_hcp=True` was explicitly passed; cascading callers defaulted to `use_existing_hcp=False` and would silently blow away a manually-overridden scorecard's playing handicap during an unrelated cascade. Fixed by checking the override unconditionally, before the `use_existing_hcp` flag — aligns the recalc path with `_process_scores`'s existing "manual overrides take precedence" intent.

**Admin UI:**
- New route `GET/POST /handicap/rebuild` (`rebuild_timeline`). GET applies the rebuild against the live connection and rolls back (`db.rollback()`) — a true preview, not a separate simulation path, so what's shown is exactly what POST commits.
- New template `templates/handicap/rebuild_timeline.html` (status table + Apply button, modeled on `admin/run_migrations.html`'s preview/apply pattern).
- New "More Tools" entry in `admin/season.html`: 🛠 Rebuild Handicap Timeline.
- Existing `↺ Recalc Handicaps` button (`recalc_season`, season-scoped, handicap-only) deliberately left unchanged — distinct, narrower legacy tool, not replaced.

### Decisions Made
- Two-phase design (per-player index rebuild, then league-wide round re-score) chosen because `handicap_index` is computed purely from a player's own gross differentials, independent of other players or playing handicap — keeps date-correctness and cross-player consistency without code duplication.
- Scoped the rewire to the 5 call sites with an actual broken cascade, not all 9+ callers of handicap recalc functions in the codebase — narrower blast radius given no live-DB access to test against in this environment.
- Preview-via-rollback instead of a separate dry-run code path, to guarantee the preview can never drift from what Apply actually does.

### Blockers / Issues
- No live DB connectivity in this environment (sandboxed, no credentials) — all work is code-review-level; the actual rebuild against production data must be triggered by @user via Admin → More Tools → 🛠 Rebuild Handicap Timeline.
- Syntax-validated (`py_compile`) all three touched files; could not run the rebuild live or verify against real data (e.g. the originally-reported matchup 93, or a multi-season/override player).
