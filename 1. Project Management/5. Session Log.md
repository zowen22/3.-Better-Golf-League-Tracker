## 2026-07-03 ‚Äî Crossing-Round Handicap Simplification (Superseding `d8da205`)

*Status: `COMPLETED`*
*Type: `Manual`*

### What Was Done
User asked what a "crossing round" was; my explanation included two factual errors ‚Äî claiming a "formula loop" justified the `d8da205` self-only-temp-handicap mechanism for the crossing round, and claiming pre-eligibility rounds "never use their own score." User correctly pushed back on both: handicap index is computed purely from gross-vs-par differentials, never from playing handicap/strokes/net, so there's no circularity in a round using its own freshly-computed index as its own entering handicap; and pre-eligibility rounds do in fact use their own round's differential to set their own temp handicap (`rebuild_player_handicap_timeline()`'s `else` branch), directly contradicting the second claim.

Re-derived the actual (weaker) justification for `d8da205`'s original design ‚Äî a golf-handicapping fairness convention ("know your handicap before you play"), not a math constraint ‚Äî and presented two options: keep `d8da205`'s mechanism, or simplify by letting the crossing round use its own real average directly. User chose simplify.

**Implementation** (`app/routes/handicap.py`, `rebuild_player_handicap_timeline()`):
- Replaced the per-round `entering_by_round[rnd['round_id']] = entering` write at the top of the loop with a local snapshot `entering_before_this_round = entering`, taken before that round's own eligibility/index computation.
- Removed the `if is_crossing_round:` self-only-temp-handicap block that previously lived inside the `elif new_index is not None:` branch ‚Äî that branch is now just the `handicap_history` INSERT + `entering = new_index`, nothing else.
- Added one line at the end of the per-round loop body: `entering_by_round[rnd['round_id']] = entering if is_crossing_round else entering_before_this_round` ‚Äî every round uses the prior-rounds-only snapshot except the crossing round, which uses the value just resolved for itself (real average, or an anchor override if present).
- `temp_ph_by_round` is now populated only by the genuine pre-eligibility `else` branch, matching pre-`d8da205` semantics for that dict.
- No changes needed to `_recalc_single_round` or `rebuild_league_handicaps_and_scores` ‚Äî the crossing round now simply falls through `_recalc_single_round`'s standard `current_handicap()`/`calc_playing_handicap()` path, same as any post-eligibility round.

**Verified end-to-end**: dropped/recreated the local dev DB, re-ran `seed_dev_db.py`, queried `handicap_history` and `scorecards` directly via `psql`. Sub player 21's round 5 (the crossing round, `min_rounds_for_handicap=2`) now shows a real, unmarked `handicap_index` of 18 (no `PRE_ELIGIBILITY_MARKER_PREFIX`) that correctly propagates to a live `handicap_at_time_of_play` of 16 (18 √ó the default 90% handicap_percent, rounded) ‚Äî no longer stuck at a stale/zero value. Round 4 (genuinely pre-eligibility) unaffected: `handicap_index` and `handicap_at_time_of_play` both 18, matching its pre-existing temp-handicap behavior.

Marked `Audits/2026-07-03-crossing-round-marker.md` **OBSOLETE** ‚Äî its entire premise (a crossing round whose Playing Hcp diverges from its Index, needing a display marker to explain the divergence) no longer exists; the crossing round's playing handicap is now derived the same way as every other round's.

### Decisions Made
- Simplification over persistence: no new schema, no new marker value, no divergence to explain. The crossing round is no longer a special case in the data model at all ‚Äî only the one-line `entering_by_round` assignment differs from every other round, and that's an internal implementation detail, not user-visible behavior needing a UI marker.
- Accepted both user corrections directly rather than defending the original (wrong) reasoning ‚Äî re-derived the real (weaker, convention-based) justification for the old design before presenting options, so the decision was made on accurate premises.

### Blockers / Issues
None. This retroactively changes stored playing-handicap/net-score/match-points data for any crossing round league-wide once `POST /handicap/rebuild` is run ‚Äî same retroactive-impact profile `d8da205` already had, just a different (now-correct) resulting value. No new user action needed beyond the rebuild admins already know to run after handicap-engine changes.

**Six real, previously-latent bugs found and fixed** getting this to actually run end-to-end (none were reachable via `py_compile`/Jinja-parse ‚Äî all required a genuinely fresh Postgres DB, which had apparently never been exercised before, only production Supabase with years of accumulated manual state):
1. `init_db.py _seed_if_empty()` used the raw SQLite connection instead of a cursor ‚Äî crashed on the very first empty-DB check (`Connection.fetchone()` doesn't exist).
2. `schema_postgres.sql`: circular FK between `courses.default_tee_id` and `tees` ‚Äî moved to a post-creation, idempotency-guarded `ALTER TABLE`.
3. Two `DATE_TRUNC('month', requested_at)` indexes on a `TIMESTAMPTZ` column failed ("functions in index expression must be marked IMMUTABLE" ‚Äî the function is STABLE, not IMMUTABLE, on a tz-aware column). Removed both.
4. The additive-migrations loop had no per-file transaction isolation ‚Äî one failure (the DATE_TRUNC one) poisoned every migration after it silently. Fixed with a savepoint per migration.
5. **5 migration files existed but were never registered** in the additive list ‚Äî same failure class as the WP3.17/WP3.18 incidents, newly discovered instances: `add_is_sub.sql`, `add_absence_sub_name.sql`, `add_round_reflections.sql`, `add_scorecard_hcp_override.sql` (**`scorecards.hcp_manually_overridden`** ‚Äî missing entirely on a fresh DB, despite being load-bearing for this whole session's absence/override work), `nullable_absence_round_id.sql`. The other 9 unregistered files were verified genuinely redundant and left alone with an explanatory comment.
6. **`rebuild_league_handicaps_and_scores()`'s Phase 2 silently skipped every round in a season with no `league_settings` row**, instead of falling back to defaults like every other call site. Real production risk (`seasons.create()` never inserts a settings row ‚Äî only the first Admin Settings save does), not just a seed-script artifact ‚Äî caught because a fresh sub's pre-eligibility round stayed at its naive initial-save handicap (0) instead of correcting to 18 after a full rebuild reported `rounds_changed: 0`.

All 6 fixes verified end-to-end: dropped and recreated the local dev DB from scratch, re-ran the seed script clean, confirmed via direct SQL query and a live Playwright screenshot that the previously-wrong value (a new sub's round-1 playing handicap) now correctly shows 18 instead of 0.

### Decisions Made
- Postgres over SQLite for local dev, despite more setup ‚Äî the app's entire query surface (casts, `ON CONFLICT`, `RETURNING`, Postgres-strict `GROUP BY`) is written for Postgres semantics; patching SQLite compatibility would mean re-litigating the whole Postgres migration in reverse.
- Left 9 of 14 unregistered migration files alone (verified redundant) rather than registering everything found ‚Äî registering an already-redundant migration just adds needless re-execution on every startup.
- `seed_dev_db.py` drives the real Flask route (test client) rather than raw SQL inserts for scorecards/hole_scores, so it doubles as a smoke test of the actual scoring pipeline, not just a data fixture.

## 2026-07-02 ‚Äî Ghost Score Correctness: Edit Scores Fix, Matrix Atomicity, League-wide Stat Audit

*Status: `COMPLETED`*
*Type: `Manual` (planned via ExitPlanMode, then implemented)*

### What Was Done

User reported: Wil & Rosie, both absent (ghost-scored) and both handicap-overridden in Week 5, showed a Net far below par instead of exactly par. Root-caused via direct code reading + one Explore agent trace (three hypotheses checked and ruled out ‚Äî `_recalc_single_round`/`_process_scores` are both internally consistent) before finding the actual cause in a third path.

**Primary bug ‚Äî `admin.py` "Edit Scores" tool had no concept of ghost/absent scorecards.** `edit_scores()` pre-filled every scorecard's stored gross ‚Äî including ghosts ‚Äî into an editable form input with no visual distinction. `_save_edited_scores()` then recomputed `net_score = gross - strokes_on_hole(CURRENT handicap)` uniformly for every player, using whatever handicap is stored *right now*. If that gross was a ghost value synthesized under an older handicap (before a since-applied override), any Edit Scores save on that round ‚Äî even one only fixing a teammate's score ‚Äî silently locked in a corrupted, too-low net. Fixed: `_save_edited_scores` now regenerates a ghost's gross fresh from the current handicap (`par + strokes_on_hole`) and sets net directly to `par` (the established ghost convention used everywhere else), ignoring whatever was submitted/pre-filled. `edit_scores.html` now shows ghost rows as non-editable with a "GHOST" badge instead of presenting inputs whose edits get silently discarded.

**Likely trigger ‚Äî Handicap Matrix override tool's silent failure mode.** `matrix_update()` committed the `scorecards.handicap_at_time_of_play` override in its own transaction *before* attempting to rebuild/resync scores via `rebuild_league_handicaps_and_scores()`; that call was wrapped in a bare try/except that swallowed any exception into an unused `recalc_errors` field, never rolled back, and the frontend never read `recalc_errors` (only checked `data.ok`, which was always `true`). A failed resync left a round's handicap changed without its ghost `hole_scores` regenerated to match ‚Äî exactly the precondition Edit Scores then corrupts. Fixed: override + resync now commit together in one transaction (or not at all on failure, with an explicit `db.rollback()` so a connection with an open transaction never returns to the pool), and the admin now sees a visible alert if the resync failed.

**League-wide stat leakage audit** (user's explicit follow-up: "ensure ghost scores don't count toward stats in any way"). Launched 3 parallel Explore agents to audit every player-facing stat computation in `reports.py`, `records.py`, `my_stats.py`, `players.py`, `standings.py`, `api.py`, `schedule.py`, `main.py` for missing `is_absent = 0` filters. Most of the codebase already does this correctly (established pattern, documented in Tech Reference's "Absence / Ghost-Score Handling" section) ‚Äî found and fixed 15 call sites that were missed: `reports.py` season summary (rounds/avg/best gross), `my_stats.py` (rounds-played count + "Recent Rounds" widget), `players.py` (profile "Rounds Played" card + season breakdown), `standings.py` (5 spots ‚Äî individual scoring stats, Low Round award, flight standings, scorecards page avg, eagle/birdie awards), `api.py` (3 spots ‚Äî mobile low-gross leaders, season records, handicap-detail round history), `schedule.py` (3 spots ‚Äî weekly low-gross/birdie/eagle leaders). Points/W-L queries deliberately left unfiltered per the existing `absence_overall_point_policy` design.

## 2026-06-30 ‚Äî Chronological Handicap Rebuild Engine

*Status: `COMPLETED`*
*Type: `Manual`*

### What Was Done

**Root cause analysis (two related bugs):**
- Bug 1: editing/resaving a past round's scorecard overwrote that round's own historical handicap with the player's CURRENT handicap, instead of the handicap that was correct as of that round's date.
- Bug 2: the existing "cascade forward" mechanism (meant to propagate a correction into later rounds) was also broken ‚Äî it flattened every future round to a single "latest" handicap value instead of correctly re-deriving a chronological progression.
- Root cause of both: every recalc path derived handicaps via "latest `handicap_history` row right now" queries, never via true date-ordered replay.

**New chronological-replay engine (`routes/handicap.py`):**
- `rebuild_player_handicap_timeline(db, player_id, league_id)` ‚Äî Phase 1, per-player. Walks all of a player's completed rounds oldest‚Üínewest; for each round, looks up that round's own season's settings (window, drop counts, %, max, carry-across); computes the windowed/dropped/averaged differential pool using only rounds up to and including the current one (correctly date-bounded by construction of the walk, no date-range SQL needed); respects manual-override anchors (`handicap_history.is_manual_override=1` keyed by `trigger_round_id`) by substituting the anchor's value as the entering handicap for the next round; deletes and reinserts all auto-computed rows with `calculated_date` set to the round's actual date. Returns `{round_id: entering_handicap_index}`.
- `rebuild_league_handicaps_and_scores(db, league_id)` ‚Äî Phase 2, league-wide orchestrator. Runs Phase 1 for every player, merges into a `{round_id: {player_id: raw_index}}` lookup, then walks every completed non-bye matchup chronologically, re-scoring each (net scores, A/B roles, match points) using each player's point-in-time handicap instead of "whatever's latest". Does not commit ‚Äî caller controls the transaction, which enables an apply-then-rollback preview.

**Wired into the 5 call sites that actually cascade:**
- `_process_scores` (scores.py) ‚Äî replaces the old two-step "recalc handicap loop" + "`_recalc_future_rounds` cascade" with one call to the new engine.
- `matrix_update`, `clear_scorecard_overrides`, `clear_handicap_override` (handicap.py).
- Settings-save cascade (admin.py) ‚Äî now also recomputes the underlying index using just-changed settings, not just re-scoring with stale `handicap_history` values.
- Deliberately **not** rewired: `score_import.py`, `self_report.py`, `api.py` (4 sites), `migration.py` ‚Äî these call the legacy `recalc_handicap_for_player` for a one-off recalc with no cascade, so they don't exhibit the bug. Documented as a known gap in `handicap.py`'s module docstring rather than silently left unflagged.
- Deleted `_recalc_future_rounds` (scores.py) as dead code once its 3 callers were rewired.

**Latent bug fixed as a side effect:**
- `_recalc_single_round`'s playing-handicap resolution only respected `hcp_manually_overridden` when `use_existing_hcp=True` was explicitly passed; cascading callers defaulted to `use_existing_hcp=False` and would silently blow away a manually-overridden scorecard's playing handicap during an unrelated cascade. Fixed by checking the override unconditionally, before the `use_existing_hcp` flag ‚Äî aligns the recalc path with `_process_scores`'s existing "manual overrides take precedence" intent.

**Admin UI:**
- New route `GET/POST /handicap/rebuild` (`rebuild_timeline`). GET applies the rebuild against the live connection and rolls back (`db.rollback()`) ‚Äî a true preview, not a separate simulation path, so what's shown is exactly what POST commits.
- New template `templates/handicap/rebuild_timeline.html` (status table + Apply button, modeled on `admin/run_migrations.html`'s preview/apply pattern).
- New "More Tools" entry in `admin/season.html`: üõÝ Rebuild Handicap Timeline.
- Existing `‚Ü∫ Recalc Handicaps` button (`recalc_season`, season-scoped, handicap-only) deliberately left unchanged ‚Äî distinct, narrower legacy tool, not replaced.

### Decisions Made
- Two-phase design (per-player index rebuild, then league-wide round re-score) chosen because `handicap_index` is computed purely from a player's own gross differentials, independent of other players or playing handicap ‚Äî keeps date-correctness and cross-player consistency without code duplication.
- Scoped the rewire to the 5 call sites with an actual broken cascade, not all 9+ callers of handicap recalc functions in the codebase ‚Äî narrower blast radius given no live-DB access to test against in this environment.
- Preview-via-rollback instead of a separate dry-run code path, to guarantee the preview can never drift from what Apply actually does.

### Blockers / Issues
- No live DB connectivity in this environment (sandboxed, no credentials) ‚Äî all work is code-review-level; the actual rebuild against production data must be triggered by @user via Admin ‚Üí More Tools ‚Üí üõÝ Rebuild Handicap Timeline.
- Syntax-validated (`py_compile`) all three touched files; could not run the rebuild live or verify against real data (e.g. the originally-reported matchup 93, or a multi-season/override player).
