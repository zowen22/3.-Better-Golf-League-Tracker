# Project Overview

*Generated from Problem Statement on: 2026-06-11*

## Summary

BetterGolfLeagueTracker is a self-hosted, multi-tenant golf league management web app built in Python/Flask. It targets full feature parity with golfleaguetracker.com and is currently deployed on Render (Supabase Postgres backend) at https://bettergolfleaguetracker-oceb.onrender.com. The long-term goal is a paid SaaS platform.

## Scope

### In:
- Full league lifecycle: seasons, teams, schedules, scores, standings, playoffs, skins
- Multi-tenant isolation (each league fully independent)
- Admin panel + member views + dual auth (shared league password + individual accounts)
- Self-hosted and cloud-hosted deployment paths
- PostgreSQL backend (migrated from SQLite)

### Out:
- Native iOS/Android apps (mobile web only; iOS app is future backlog)
- USGA slope/rating handicap system (par-based differential by design)
- Multi-league user accounts (deferred)
- Extended content features (image gallery, document library)

## Decisions Log

| Date       | Decision | Rationale |
|------------|----------|-----------|
| 2026-05-26 | Par-based handicap differential (not USGA slope/rating) | Lower complexity; appropriate for recreational leagues |
| 2026-05-26 | Raw SQL with psycopg2, no SQLAlchemy ORM | Minimize rewrite scope; direct control over queries |
| 2026-05-26 | `RealDictCursor` for psycopg2 | Preserves `row['column']` access pattern from sqlite3.Row — zero template changes |
| 2026-05-26 | `psycopg2-binary` over psycopg3 | No C deps to compile; can upgrade later |
| 2026-06-11 | Deploy on Render + Supabase | Free tier sufficient for testing; Supabase connection pooler on port 6543 required (direct port 5432 is IPv6-only on Render) |
| 2026-06-11 | `postgres-migration` branch is primary working branch | Main branch is SQLite-era code; all active work is on postgres-migration |
| 2026-05-26 | Scores carry across seasons by default | Preserves handicap continuity for players in multi-season leagues |
| 2026-05-26 | Match play: Win=2, Tie=1 each, Loss=0 per hole + overall | Max 20 pts/player/matchup (9 holes × 2 + 2 overall) |
| 2026-05-26 | Bye weeks: no points awarded | Keeps standings fair; avoids inflating totals for easy schedules |
| 2026-06-20 | `/api/v1/courses` returns tees grouped by color (Option B) | iOS needs to show one tee per color in picker and resolve to tee_id via matchup's nine at submit time. Option A (single M-rep per color) would require iOS to infer the nine; Option B (color → nines dict) gives the app everything without extra lookups. |
| 2026-06-17 | iOS Score Entry tab: admin-only for now | Considered opening OCR photo capture to all members as a self-report shortcut, but no clean validation path without storing photos server-side. Forcing camera-roll saves is bypassable via permission denial. Admin remains the score uploader. Revisit if photo storage becomes viable. |
| 2026-06-17 | iOS tab order: Schedule, Standings, Score Entry (admin), Stats, Admin Panel (admin) | Stats is a catch-all for All-Play, Weekly, Handicap Trend, Leaders, Records. Score Entry and Admin tabs hidden for non-admin roles. |
| 2026-07-15 | Pricing finalized: $99/league/season standalone, $79/league/season at 10+ leagues (volume) | Undercuts GLT's $119 (17% off) / $89 volume tier (11% off) by design; positioning leads with "no paid add-ons" since GLT charges $20-99 extra for features BGLT already bundles free — no billing infra built yet, this is the number to build against |
| 2026-07-03 | Overriding priority for every session going forward: work toward getting this product to a shippable, paid-service state | Elevates Problem Statement goal #3 ("long-term: pivot to paid SaaS platform") from a future aspiration to the active operating priority — feature-parity gaps, payment processing, and onboarding/admin infra (Phase 4: SaaS Pivot in Work Packages) should be weighed against this, not treated as someday-backlog |
| 2026-07-04 | Dedicated per-feature flows, not one overloaded grouping mechanism — deliberately diverge from GLT here | GLT funnels divisions, skins flights, playoff brackets, and standings filtering all through a single "League Groups" setting; the user finds this un-friendly (configure groups once, then must remember they silently drive five features). BGLT already leans the other way (Skins, Playoffs are already their own nav items). Standing design principle: skins flights, divisions, multi-league management, playoffs, etc. should each be their own clearly-labeled user-facing flow. Backends stay **separate** unless a shared "grouping" primitive later proves genuinely necessary — do not pre-build a general grouping system (that would recreate GLT's overloaded-mechanism problem). Applies to the skins-flights build (`Handoffs/2026-07-04-skins-flights.md`) and future divisions/multi-league work. |
