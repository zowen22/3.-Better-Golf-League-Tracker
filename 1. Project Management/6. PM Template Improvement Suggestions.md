# PM Template Improvement Suggestions

This file captures suggested improvements to the base CLAUDE.md project management template, observed during real project work. It is **not** part of the template itself — it exists only in projects that generate suggestions.

---

## How to Use This File

When a Claude session observes a gap or friction point in the PM structure, it logs a suggestion here instead of modifying CLAUDE.md directly (which is template-sourced and should stay clean).

**To apply these suggestions to the template:**
Hand this file to a separate agent with the instruction: "Apply these suggestions to the CLAUDE.md template and any associated PM file stubs." That agent should treat each suggestion as a discrete change proposal, apply judgment on what to adopt, and update the template accordingly.

---

## Suggestions

### 1. Clarify Session Start Routine

**Observed friction:** Step 1 says "Check if project files have been shared" — but files live in the repo, not shared by the user. This creates ambiguity about when to read PM files.

**Suggested change to CLAUDE.md Session Start Routine:**
Replace:
> 1. Check if project files have been shared
> 2. If yes — read all files, confirm current status and next priorities

With:
> 1. Read all `1. Project Management/` files to orient on current status and next priorities

---

### 2. Add Memory vs PM Files Rule

**Observed friction:** Claude's built-in memory system (files outside the repo at `~/.claude/projects/.../memory/`) can duplicate content that belongs in Technical Reference, and those memory files aren't committed to the repo — they get lost.

**Suggested addition to CLAUDE.md Rules:**
```
## Memory vs PM Files
The Claude memory system is for thin cross-session pointers only — not content.
All durable knowledge lives in PM files (committed to the repo):
- Architecture decisions → Decisions Log (2. Project Overview.md)
- Conventions, patterns, gotchas → 4. Technical Reference.md
Memory files should say "see Technical Reference §X" not repeat the content.
```

---

### 3. Add a Known Issues / Bug Log File

**Observed friction:** Recurring bugs (e.g. tee deduplication showing duplicates across web and iOS) got re-diagnosed in a later session because there was no issue tracker. The Session Log captures fixes but is chronological — not queryable by topic.

**Suggested new PM file:** `6. Known Issues.md` (or `Bug Log.md`)
- Lightweight table: Issue | Status | Fixed In Session | Notes
- Bugs that span sessions or risk regression get logged here
- Closed when fixed; never deleted (mark `[Resolved]`)

---

### 4. Cross-Reference Audit Findings to Work Package Tasks

**Observed friction:** CLAUDE.md lists open audits but findings aren't linked to specific WP task IDs. When a finding gets fixed, there's no clear path to mark it resolved in the audit file — the two systems drift out of sync.

**Suggested addition to Rules:**
```
- Each audit finding should reference a WP task ID (e.g. "WP5.3-P2")
- Closing the WP task → mark the audit finding [Fixed] in the same session
```

---

### 5. Archive Old Session Log Entries

**Observed friction:** Session Log grows unbounded. After many sessions it becomes slow to read at session start and noisy.

**Suggested addition to Rules:**
```
- Keep only the last 5 sessions in 5. Session Log.md
- Archive older entries to 5. Session Log Archive.md
- Archive at session end when log exceeds ~10 sessions
```

---

### 6. Track iOS / Client Work in Work Packages

**Observed friction:** iOS feature parity tasks were called out in Session Log "What's Next" notes but never added to Work Packages. They became invisible to the WP progress tracker and easy to miss at session start.

**Suggested addition to Rules:**
```
- Any "What's Next" item in the Session Log must have a corresponding WP task
  created before the session is marked COMPLETED
- Cross-platform work (iOS, web, API) should be explicit separate tasks, not bundled
```

---

### 7. Add PM Improvement Suggestion Process to CLAUDE.md

**Observed friction:** There was no instruction for where to put PM improvement ideas, so they ended up inline in CLAUDE.md (wrong place) before being moved here.

**Suggested addition to CLAUDE.md Rules:**
```
- If you observe a gap or friction point in this PM structure, log it in
  `1. Project Management/6. PM Template Improvement Suggestions.md`
  Do NOT modify CLAUDE.md itself with improvement notes — that file is template-sourced.
```
